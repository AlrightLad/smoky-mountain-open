# Section 1 — Current State

This section reports what exists in the codebase today. Facts only; interpretation follows in later sections.

## 1.1 What roles currently exist

Two distinct, non-unified role concepts exist today:

### 1.1.a — `role` field on the member doc (`members/{uid}.role`)

Used for **platform-wide** authorization. Possible values in the code:

- `"commissioner"` — assigned once, to the first user who registers with invite code `"FOUNDING-FOUR"` (`src/core/firebase.js:300-302`). All other code paths default new members to `"member"` (`firebase.js:503, 534`). A local-fallback path sets `"commissioner"` for unauthenticated local users (`firebase.js:542`).
- `"member"` — the default for every registered user that isn't the first FOUNDING-FOUR user (`firebase.js:358` for new profile, `firebase.js:333` for claimed-profile path).
- `"suspended"` — set by the admin panel (`src/pages/admin.js:328`) with companion fields `suspendedAt`, `suspendedUntil`, `suspendedBy`, `suspendedReason`.
- `"removed"` — set by the admin panel (`admin.js:372`) with companion fields `removedAt`, `removedBy`, `removedReason`.

No `"admin"` value exists on the member doc. No `"founder"` value exists.

### 1.1.b — League doc fields (`leagues/{leagueId}`)

Used for **per-league** authorization. Created at league creation in `src/pages/leagues.js:132-135`:

- `commissioner` — a single uid string. The league creator becomes the league commissioner.
- `admins` — an array of uids. The creator is seeded into their own `admins` array by default.
- `memberUids` — an array of uids representing everyone currently in the league.

### 1.1.c — Founding-four flags

- `isFoundingFour` (boolean) on member doc — true for members who registered with invite code `"FOUNDING-FOUR"` (`firebase.js:339`).
- `founding` (boolean) on member doc — same value as `isFoundingFour`, used in some display paths (`members.js:115, 139, 190, 300, 450`).
- `badges: ["founder"]` on member doc — seeded for founding-four members (`firebase.js:341`).
- League doc `badge: "founding"` — exactly one league carries this; it's hard-gated as non-deletable in `leagues.js:258, 470`.

None of these are a "role" per se — they're flags and badges. There is no `role: "founder"` path.

### 1.1.d — Firestore rules role check

`firestore.rules:6` defines:

```
function isCommissioner() {
  return isAuth() && get(/databases/$(database)/documents/members/$(uid())).data.role == "commissioner";
}
```

Note: this resolves against the member doc's platform-wide `role`. It does **not** consult the league doc's `commissioner` field. As a consequence, the per-league `commissioner` field carries no elevated rules permissions — it's a client-side signal only.

## 1.2 What each existing role can do

### Platform Commissioner (`member.role == "commissioner"`) — enforced in Firestore rules

- Update any member doc (`firestore.rules:12`).
- Delete any member doc (`firestore.rules:13`).
- Update or delete any round (`firestore.rules:20`).
- Update or delete any chat message in any league (`firestore.rules:36`).
- Update or delete any invite (`firestore.rules:50`).
- Write config docs (`firestore.rules:54`).
- Update any wager regardless of participant (`firestore.rules:78`).
- Delete any bounty (`firestore.rules:87`).
- Delete any tee time (`firestore.rules:110`).
- Delete any scramble team (`firestore.rules:114`).
- Read errors, feature_requests, reports (`firestore.rules:134, 136, 137`).

### Platform Commissioner — enforced in client code only

- Access the admin panel (`src/pages/admin.js:3`).
- Suspend a member (`admin.js:313-343`).
- Remove a member from the platform (`admin.js:364-385`).
- Reinstate a member (`admin.js:387-403`).
- Bulk-generate invite codes (`admin.js:29-35`).
- Run full diagnostic and data-recovery scans (`admin.js:62, 69`).
- Override attestations on an event round (`src/core/sync.js:887`).
- Delete cancelled or past tee times (`src/pages/teetimes.js:128, 212`).
- Delete party games (`src/pages/partygames.js:53, 142`).
- Edit any scorecard when the trip is not closed (`src/pages/scorecard.js:31, 221-222, 290, 319`).
- Unlimited invite quota (`firebase.js:336, 360`; `members.js:1618`; `invite.js:4`).
- "The Commissioner" title is reserved for this role in the shop (`shop.js:90`).
- Unique achievement "the_commish" awarded automatically (`data.js:1345`, +500 XP).

### League Commissioner (`league.commissioner == uid`) — client-side only

- Access the league detail page's settings section (`leagues.js:237, 247`).
- Toggle league visibility public/private (`leagues.js:429`).
- Toggle join-request approval requirement (`leagues.js:441`).
- Regenerate the league's invite code (`leagues.js:453`).
- Attempt to delete the league if it isn't the founding league (`leagues.js:462, 470`). Note: this will fail in production because `firestore.rules:71` globally denies league deletion — this is latent broken code, not a working capability.
- Promote or demote league admins (`leagues.js:354, 418`).
- Approve or deny join requests (`leagues.js:388, 408`).
- Update the league doc itself (`firestore.rules:70` — scoped to `resource.data.commissioner == uid()`).

### League Admin (`uid in league.admins`) — client-side only

- View the league member list (`leagues.js:234, 240`).
- View pending join requests (`leagues.js:243, 244`).
- Approve or deny join requests (`leagues.js:380, 381`).

League admins **cannot** do any of these, even though the UI sometimes suggests otherwise:

- They cannot change league settings (`leagues.js:247` gates settings on `isComm` only).
- They cannot promote other admins (`leagues.js:354` restricts the admin-toggle button to the commissioner).
- They cannot regenerate invite codes.
- They cannot delete the league.
- They cannot directly write the league doc (`firestore.rules:70` only allows writes where `resource.data.commissioner == uid()`).

### Member (`member.role == "member"`, default)

- Log own rounds, range sessions, DMs, chat, photos, tee times.
- Update or delete own content only.
- Invite up to `maxInvites` (default 3) new users.
- Read the entire corpus of data that `isAuth()` allows.

### Suspended (`member.role == "suspended"`)

- **Functionally identical to `member` in Firestore rules.** No rule references `"suspended"`. The check is cosmetic and exists only in the admin panel display (`admin.js:245, 249, 270`) and the member card display (`members.js:160`).
- Companion field `suspendedUntil` is set, but nothing reads it to block access — no middleware enforces the suspension on read/write paths.

### Removed (`member.role == "removed"`)

- Hidden from rosters and lookups in multiple client paths: `data.js:140`, `findplayers.js:33`, `dms.js:24`, `partygames.js:79`, `rounds.js:315`, `members.js:61`, `teetimes.js:177`.
- **No Firestore rules reference `"removed"`.** A removed user whose auth session is still valid can continue to read anything `isAuth()` allows and write anything their own-uid checks allow (own rounds, own chat, etc.). Removal is a client-side veil, not a server-side gate.

## 1.3 What happens when a user has no active league

### The core fallback

`src/core/sync.js:4-7` defines:

```
function getActiveLeague() {
  if (currentProfile && currentProfile.activeLeague) return currentProfile.activeLeague;
  return "the-parbaughs";
}
```

Any user whose `currentProfile.activeLeague` is empty string, undefined, or null silently falls back to `"the-parbaughs"` for every league-scoped read and write. This makes the leagueless state **effectively invisible**: the user is treated as if they were browsing the founding league.

### Can a user actually be leagueless?

**At registration:** No. `src/core/firebase.js:249` requires an invite code. All working invite codes carry a `leagueId` (typically `"the-parbaughs"`, assigned at `firebase.js:362` via `_invLeague`). The FOUNDING-FOUR code also lands users in `"the-parbaughs"`. There is no registration path that produces a user with `leagues: []`.

**After registration:** Yes, in one edge case. The league delete flow at `leagues.js:462-491` removes the league from every member's `leagues` array. If the deleted league was a member's **only** league, they end up with `leagues: []` and `activeLeague: ""` (set at `leagues.js:483`). Nothing in the app currently surfaces this state — they'd be shunted back to `"the-parbaughs"` by `getActiveLeague()` but they would not be a member of it, so:

- League-scoped reads with `.where("leagueId", "==", "the-parbaughs")` would return data they can see but have no place in.
- League-scoped writes would tag new rows with `leagueId: "the-parbaughs"` and inject foreign data into the founding league.

This is a silent bug, not an intentional design.

### What a leagueless user would see today

- **Home page:** renders normally. The home page doesn't explicitly check league membership, so it would show "the-parbaughs" content.
- **Leagues page (`leagues.js:20-23`):** has a visible "No Leagues Yet" empty state, but only if you navigate there.
- **Chat, feed, tee times, rounds:** populate with the-parbaughs' data.
- **No modal, banner, or redirect** prompts the user to join or create a league.

## 1.4 Missing governance actions

Grep-verified absent from the codebase:

- **`remove-member`** — per-league member removal. Only platform-wide `removeMemberAdmin` exists (`admin.js:364`). There is no "kick from league" action that preserves the user's other leagues.
- **`ban-user`** — there is no enforcement layer preventing a removed user from returning via a new email. Removal flips a flag, nothing more.
- **`leave-league`** — no "Leave this league" button or function. A user who joins a league has no self-service way out.
- **`transfer-commissioner`** — no function exists to transfer the `league.commissioner` uid from one user to another.
- **`founder`** — no distinct role, badge visual, or capability. The `badges: ["founder"]` array tag is set (`firebase.js:341`) but never rendered anywhere.
- **Appeal flow** — no surface exists for a suspended or removed user to submit an appeal.
- **Bounty expiry refund** — called out in CLAUDE.md's Known TODOs but not implemented.
- **Per-league admin capabilities beyond view + approve-requests** — admins can't kick, can't promote, can't edit anything.

# Section 2 — Proposed Role Hierarchy

Below is a proposed full hierarchy. Each role is specified by: scope, assignment, revocation, uniqueness, and what fields on which doc(s) carry it. Implementation details (field names, storage) are illustrative — Stage 2 will finalize.

## 2.1 Platform Founder (Zach)

- **Scope:** platform-wide, cross-league, non-transferable.
- **Held by:** exactly one person — Zach. The role is conceptually tied to the platform itself, not to any league.
- **Assignment:** hardcoded at platform inception. Cannot be self-assigned or earned.
- **Revocation:** never, except by Zach himself. See Section 6.3 for the account-deletion edge case.
- **Uniqueness:** singleton.
- **Capabilities:** every capability in the matrix. The Founder is the platform's ultimate backstop for support, moderation appeals, and cross-league disputes.
- **Distinct from:** Commissioner of the founding league, even though the same human holds both. Zach is Founder because he built the platform; he is Commissioner of "the-parbaughs" because he runs that league. Losing the latter should not lose the former.

**RESOLVED — 2.1.a.** Should the Founder role live on a distinct field (e.g. `platformRole: "founder"`) rather than being inferred from uid equality with a hardcoded value? Recommendation: explicit field, so a database migration to a new Firebase project doesn't break it.

**>>> RESOLVED:** Explicit `platformRole` field. No hardcoded UID checks. Migration-safe, consistent with other role patterns, inspectable.

## 2.2 League Commissioner

- **Scope:** per-league.
- **Held by:** one member per league at any time.
- **Assignment:** default to the league's creator. Transferable via an explicit flow (see 2.2.a below).
- **Revocation:** occurs on transfer. A commissioner cannot have the role revoked by anyone other than the Founder (exceptional support intervention) or by voluntarily transferring it.
- **Uniqueness:** one per league.
- **Capabilities:** full control of their league — admins, members, settings, invite codes, deletion (with guardrails).
- **Cross-league:** holding commissioner in League A grants nothing in League B.

**RESOLVED — 2.2.a.** Should League Commissioner be transferrable at all?

- **Option A — Locked to creator.** Simpler. Predictable: if you start a league, you run it, period. If you want out, you delete the league.
- **Option B — Transferrable with a two-step handshake.** Commissioner nominates a successor; successor accepts; transfer is logged. Solves "I want to step back but keep the league alive."
- **Option C — Transferrable AND co-commissionable (two commissioners simultaneously).** Most flexible. Adds conflict-resolution complexity: who wins if they disagree?

**Recommendation: Option B.** A golf league is a long-lived social object; people graduate, move, get too busy. Forcing deletion when the commissioner wants to step down kills history. Two-commissioner mode (Option C) invites the dispute problems the Founder role exists to avoid — one decision-maker per league is healthier.

**>>> RESOLVED:** Yes — handshake flow (current commissioner nominates, nominee accepts). Future refinement: add a 24–48h cooling-off window where the outgoing commissioner can revoke the transfer. Cooling-off is not required for v8.0 launch — it's a v8.1+ refinement.

**RESOLVED — 2.2.b.** If B is chosen: can the Founder force a transfer in exceptional cases (abandoned league, hostile commissioner refusing to transfer, abusive commissioner harming members)?

**Recommendation:** yes, but only through a documented escalation — a member files a formal "request commissioner override," the Founder reviews, and if warranted, transfers. This is the nuclear option and should rarely fire.

**>>> RESOLVED:** Yes, via documented escalation with audit trail. Force-transfers create a visible audit record showing timestamp, old commissioner, new commissioner, and stated reason. The record is visible to affected league members after the fact so the action can't happen silently.

## 2.3 League Admin

- **Scope:** per-league.
- **Held by:** zero or more members per league.
- **Assignment:** by the League Commissioner. A Commissioner can promote any current league member to admin.
- **Revocation:** by the League Commissioner. Admin status is also revoked if the member leaves or is removed from the league.
- **Uniqueness:** multiple allowed.
- **Capabilities:** a trusted lieutenant. Can approve/deny join requests, remove members (except the commissioner and other admins), moderate chat in that league. Cannot promote other admins, cannot delete the league, cannot change invite codes or visibility.
- **Cross-league:** same rule as commissioner — admin in League A grants nothing in League B.

**RESOLVED — 2.3.a.** Should leagues require a minimum number of admins?

- **Option A — No minimum.** A sole commissioner is legal. Simplest.
- **Option B — Require ≥1 admin before a league can grow past N members** (e.g., require 1 admin once the league hits 25 members).

**Recommendation: Option A at launch.** Adding required-admin thresholds is a complexity tax on small leagues. The 17-person Parbaughs runs fine with one commissioner and zero admins. Revisit if we see actual scaling pain.

**>>> RESOLVED:** No minimum. Leagues that grow add admins organically.

## 2.4 League Member

- **Scope:** per-league.
- **Held by:** anyone who has joined the league via invite code or approved request.
- **Assignment:** on join. No explicit action to grant — joining makes you a member.
- **Revocation:** voluntary via `leaveLeague`; involuntary via kick by a commissioner or admin.
- **Uniqueness:** N/A.
- **Capabilities:** standard participation — log rounds, chat, DM, RSVP tee times, create bounties and wagers, etc.
- **Cross-league:** membership travels with `leagues[]` array. A user can be member of many leagues.

## 2.5 Leagueless User

- **Scope:** platform-wide, but with no active league context.
- **Held by:** a user who is authenticated but in zero leagues.
- **Assignment:** this is a transient state, not an assigned role. See Section 5 for whether this state should exist at all.
- **Revocation:** resolved the moment the user joins or creates a league.
- **Uniqueness:** N/A.
- **Capabilities:** determined by Section 5's decision on whether leagueless is a permitted state.

## 2.6 Global vs. per-league role storage — principle

Two fundamentally different axes exist, and today they're conflated on a single `role` field. The proposed split:

- **Platform role** — `platformRole` on the member doc, values: `"founder"`, `"user"`, `"suspended"`, `"banned"`. This replaces the current overloaded `role` field.
- **League roles** — the `commissioner` singleton field and `admins` array on the league doc (these already exist; v8 keeps them).

The `"suspended"` and `"removed"` values currently on the `role` field are platform states, not roles — they belong on `platformRole`. Once a member is platform-suspended, they're suspended in every league simultaneously. This also fixes a latent bug: today, suspending a commissioner overwrites their role to `"suspended"` and erases their commissioner status permanently — the reinstate flow resets to `"member"`, not the original role.

**RESOLVED — 2.6.a.** Do we split `role` into `platformRole` + per-league role tables immediately (Stage 2 foundation work) or keep them conflated and paper over in rules?

**Recommendation:** split immediately. The current conflation is the root cause of both the suspended-commissioner bug and the absent-capability pattern. A one-time migration in v8.0 is cheaper than shimming around it forever.

**>>> RESOLVED:** Yes — split immediately in v8.0.0. Fixes the latent suspended-commissioner bug structurally.

⁶ **RESOLVED — 3.4.a.** Can leagueless users DM anyone? Three options:

| Platform-ban a user | ✓ | ✗ | ✗ | ✗ |
| Override any league's governance | ✓ | ✗ | ✗ | ✗ |
| Read any league's data for support | ✓ | ✗ | ✗ | ✗ |
| Force-transfer a league commissioner | ✓ | ✗ | ✗ | ✗ |
| Force-delete a league (e.g., abandoned) | ✓ | ✗ | ✗ | ✗ |
| Handle ban appeals | ✓ | ✗ | ✗ | ✗ |
| Rotate GolfCourseAPI and FCM keys | ✓ | ✗ | ✗ | ✗ |
| Set feature flags / kill-switches | ✓ | ✗ | ✗ | ✗ |

**RESOLVED — 3.5.a.** "Read any league's data for support" is a legitimate capability (a commissioner DMs "we have a bug" — Founder needs to see the data to help). But it's also a privacy risk: Founder can read private DMs in any league. Three options:

- **Option A — Founder has unrestricted read on all data.** Simplest; matches current reality (Zach sees everything today).
- **Option B — Founder has unrestricted read except for DMs, which require the user's explicit opt-in for a support session.** Privacy-preserving but complicates support flows.
- **Option C — Founder has unrestricted read always, but every access is logged to an audit trail visible to the affected user.** Middle ground.

**>>> RESOLVED:** Option C from day one. Unrestricted Founder read across all league data including private content, with an audit log visible to affected league members. Only private-content reads log entries (DMs, private chat, moderation-sensitive records). Aggregate stats and public-content reads don't log — they'd drown the signal. Audit entries include: timestamp, leagueId, dataType, reason. Trust is cheaper to build in from the start than to retrofit.

# Section 4 — Visibility Matrix

This section specifies what other users see about each role-holder. Visual language should be restrained — only the Founder and Commissioners get treatment visible across all surfaces; lesser roles are visible only where contextually relevant.

## 4.1 Visibility per role

### 4.1.a — Founder

- **Title next to name:** Always present platform-wide. Treatment TBD per 4.2.
- **Badge icon:** A single distinctive mark (shape and color TBD per 4.2).
- **Name effect / ring:** See 4.2 for the three options.
- **Where visible:** profile page, profile bar, feed posts, chat messages, member lists, DM headers, leaderboards. Effectively everywhere the user's name appears.

- **Title next to name:** Visible only within the league where the role applies. In other leagues, the user appears as a regular member.
- **Badge icon:** Small gold tag ("COMM" or a gavel icon) next to their name in league-scoped surfaces.
- **Name effect / ring:** None. Avoiding over-decoration — a per-league role shouldn't compete visually with the Founder treatment.
- **Where visible:** league detail page member list, league chat messages, league feed posts, league leaderboards. **Not** visible on global profile, cross-league DMs, or other leagues.

### 4.1.c — League Admin

- **Title next to name:** Visible only within the league where the role applies, and only in moderation-adjacent UI (member list, pending-request views).
- **Badge icon:** Small blue tag ("ADMIN"), shown next to their name in the league member list and when they perform moderation actions (e.g., "Approved by [Admin Name]").
- **Name effect / ring:** None.

### 4.1.d — Member

- No title, no badge, no ring. Standard display.

### 4.1.e — Leagueless

- No title. A "Find a League" banner shows on the home page. See Section 5.

### 4.1.f — Suspended

- A red "Suspended" tag in the member's own profile. Suspended members typically aren't rendered in others' feeds during the suspension — but when they do appear (e.g., on a historical round scoreboard), the tag signals their status.

### 4.1.g — Banned

- Not rendered in most lists (like today's "removed"). On historical records, name appears with a gray "Former Member" tag rather than the current silent-disappear behavior.

## 4.2 The Founder visual treatment

Specifically for the Founder role, three options:

### Option A — Quiet

- Reserved title "Founder" shown only on Zach's profile page.
- No special badge.
- No ring or name effect.
- No cross-surface treatment.

**Pros:** humble, understated. Matches the "built by a friend for friends" ethos.
**Cons:** invisible at scale. New users don't know who made the app or who to escalate to.

**Pros:** tasteful. Distinctive without shouting.
**Cons:** subtle enough that most users won't understand what it means without context.

### Option C — Full treatment

**Pros:** immediately legible. New users see "Founder" and understand the hierarchy.
**Cons:** loud. Could feel like a power trip at the 17-member scale.

**Recommendation:** **Option B**. The founding league is Zach's crew — they know him. At scale (1000+ users), the unique ring tells new users "this person is different" without the "Founder" word stamping itself on every interaction. If legibility becomes a problem, add the reserved title in v8.2; adding loudness is easier than removing it.

**RESOLVED — 4.2.a.** Pick one of A / B / C.

**>>> RESOLVED:** Modified Option C. The Founder visual treatment includes: "Founder" title next to name on profile, feed posts, chat, and leaderboards; unique one-of-one ring; subtle founder badge icon wherever name appears. Name color stays theme-default — no gold name text — so the treatment doesn't compete with achievement gold, champion red, or earned name effects. Loudness concern from the original recommendation is addressed by keeping name color off the list.

**RESOLVED — 4.2.b.** If Option B or C: what ring color? Candidates:

- **Iridescent / oil-slick gradient** — most distinctive, but risks looking trendy-in-a-bad-way.

**>>> RESOLVED:** Custom one-of-one. Not in the shop, not shoppable, not reservable, not earnable. Code-level gated to Zach's account only.

Ring: "Glowing opal diamond." Real opal color-shift (rainbow fire from actual opal stones, NOT cartoon rainbow) with a diamond sparkle layer on top, one tier brighter and more active than the shop's Diamond Sparkle.

Name effect: prismatic refraction. Name appears with subtle light-splitting as if viewed through a crystal — slow, organic motion, not aggressive cycling.

Implementation is craft work, not a CSS quick-hit. Ship during v8.0.1 Visibility Polish with 3–4 mockup iterations before code lands. Do NOT ship the first gradient that "kind of works." Fallback if prismatic refraction looks cheap in mockups: opal shimmer text matching the ring palette.

**>>> RESOLVED:** Option D. Role displays as a small badge icon (Admin chip, Commissioner gavel, Founder mark). Title displays as text next to the name. Both coexist because they use different visual channels.

# Section 5 — Leagueless User Experience

## 5.1 Fundamental question — can a leagueless user exist?

**RESOLVED — 5.1.a.** Should the platform allow accounts to exist outside any league?

- **Option A — No. Leagueless is forbidden.** Every registration path must land the user in at least one league. If a user's last league is deleted, auto-create a "My Leagues" shell or re-enroll them in a public default. The `getActiveLeague()` fallback becomes impossible by construction.
- **Option B — Yes, leagueless is a first-class state.** Users can register without an invite code and browse public leagues before committing. The home page shows a "Find a League" discovery experience.
- **Option C — Yes for discovery, but no for content creation.** Users can exist leagueless to browse, but cannot log rounds, chat, or post anything until they join a league.

**>>> RESOLVED:** Modified Option B. Users can exist leagueless with full access to solo and social features: register without an invite code, log rounds solo, use analytics and The Caddie, browse the course directory, add friends via handle search, DM friends, earn and spend ParCoins, and browse public leagues. League-specific features (league chat, league leaderboards, league events, league-member wagers) still require membership.

Strategic implication: PARBAUGHS is solo-first with a league layer added on top. Leagues are additive, not gating. This flows through to App Store positioning, marketing copy, and Phase 4 monetization framing. The earlier recommendation (Option C — browse-only leagueless) under-counted how much of the product is inherently personal (your rounds, your handicap, your insights) and how few of those features actually require a league.

This decision has a secondary effect on 3.1 (capability matrix): LLess users now have ✓ on logging rounds solo, viewing own stats, DMing friends, browsing courses. League-scoped rows in 3.1 (View content in own league, Log rounds in own league, Post in league chat) remain ✗ for LLess because those rows are about league context, not content creation per se.

## 5.2 On the home page, what does a leagueless user see?

Assuming Option C is chosen:

- **Top of page:** a single hero card — "You're not in a league yet" — with two primary CTAs: "Join with a code" and "Create your own league." A secondary "Browse public leagues" link below.
- **Middle of page:** a trimmed public-leagues discovery strip. Three featured leagues (curated or ranked by some metric — size, activity, geographic proximity).
- **Bottom of page:** the rest of the home page content is suppressed. No "Spotlight" (there's no league to spotlight from). No personal stat bar (stats accumulate only from played rounds, which they have none of). No feed.

**RESOLVED — 5.2.a.** Should we seed a leagueless user with any content at all — e.g., a "platform feed" of rounds from public leagues they might find interesting? Recommendation: no. Keep the onboarding funnel single-purpose: convert to a league member.

**>>> RESOLVED:** Yes — but personal-and-actionable content only. Home page shows: public leagues discovery strip (3–5 curated), friend activity feed once the user has friends, Tip of the Day, and personal stats plus The Caddie insights once rounds are logged. A persistent but non-intrusive prompt to join or create a league stays visible. Do NOT seed a firehose of random platform activity from strangers — it looks spammy and distorts the per-user signal. This updates the prior "no seeding" posture that was tied to the Option C (browse-only) leagueless model.

## 5.3 Can they log rounds? View other members?

Assuming Option C:

- **Log rounds:** no. Rounds are a league-scoped artifact. Attempting to log without a league redirects to the join/create experience.
- **View other members:** only **public** profiles of members in **public** leagues. No member list access; no DM access (per 3.4.a Option B, they retain DMs only with pre-existing friends — but leagueless-from-the-start users have none).
- **View range sessions, wagers, tee times:** all are league-scoped writes. Same redirect.
- **View global shop items:** yes. Browsing cosmetics they can earn once in a league is healthy funnel content.
- **Edit own profile:** yes. Filling out bio, handicap, home course, avatar — all normal.
- **Manage settings:** yes.

## 5.4 How do they discover leagues?

- **Browse-by-location:** if they've set a home course or location in their profile, sort public leagues by geographic proximity.
- **Browse-by-size:** "Small & tight" (5–20 members), "Medium" (21–50), "Large" (51+).
- **Invite code:** enter a code they received out-of-band.
- **Invite link:** tap a share-sheet link from a friend.
- **Search:** by league name (exact / fuzzy).

**RESOLVED — 5.4.a.** Should public leagues be rankable / filterable at launch, or is "show the first 20 public leagues" good enough until volume demands it? Recommendation: minimal sort (by recency or member count) at launch. Don't over-engineer discovery for a platform with <100 public leagues.

**>>> RESOLVED:** Minimal sort. Default is most-recent-activity. League size is visible on each card, location is shown if set, basic text search by name. No filters, no ranking algorithms at launch.

## 5.5 Empty state — no league, no rounds

The ultimate empty state. Design as a single-focus landing:

- Heading: "Your golf crew starts here."
- Sub: "Join a friend's league with a code, or start your own crew of 2+."
- Two buttons: "Join with a code" / "Start a league."
- Small link: "Browse public leagues."

No stat bar with zeros. No feed placeholder. No "0 rounds" counter. These are demotivating artifacts of the current skeletal-home logic — they say "you've failed" before the user has done anything. Replace them with forward-motion copy.

**RESOLVED — 5.5.a.** Should leagueless users see a full-screen "onboarding wizard" modal on first login, or just a quiet home page they can explore? Recommendation: a quiet home page with a prominent CTA. Wizards are annoying and imply the user is incompetent. If the CTA doesn't convert, consider a nudge notification at 24h / 7d.

**>>> RESOLVED:** No wizard. Quiet home page with one clear primary CTA, inline hints on first-use of specific features, an optional "take the tour" available in settings. Graduated nudges: soft in-app banner at 24h if still leagueless, notification at 7d, stop nudging at 14d. Nudge timing tuned post-launch based on actual conversion data.

# Section 6 — Edge Cases

Hard situations and how governance handles them. Each decision point was flagged during design; Zach's resolutions are inline under **RESOLVED** markers.

## 6.1 Commissioner wants to leave the platform permanently

They're done. They want to delete their account.

**Scenarios:**

- **Scenario A — They're commissioner of the founding league.** This is essentially Zach. Covered in 6.3.
- **Scenario B — They're commissioner of a side league with active members.** Options:
  - Block account deletion until they transfer the commissioner role. Clear message: "You're running League X — transfer it first or delete the league."
  - Allow deletion; auto-promote the longest-tenured admin, or the member with the highest member-rank (handicap? activity? tenure?) to commissioner.
  - Allow deletion; the league becomes "orphaned" and enters a grace state — see 6.2.

**RESOLVED — 6.1.a.** Which of the three? Recommendation: block account deletion until transfer or delete-league. Auto-promotion is a forced burden on someone who may not want the responsibility. Orphaned leagues are a maintenance nightmare. The block is a small friction that prevents a large mess.

**>>> RESOLVED:** Option A — block. Account deletion is blocked for any user holding Commissioner role in any league. The user must either transfer the commissioner role or delete the league before their account can be deleted. Applied per-league: a user commissioning multiple leagues must resolve each one.

- **Scenario C — They're commissioner of multiple leagues.** Same treatment as B, applied to each league.

## 6.2 League has one member (just the commissioner) and they leave

The commissioner is the sole member, and they delete their account or leave the league.

**Options:**

- **Option A — League auto-deletes.** Clean. No orphans. Destroys history.
- **Option B — League is archived.** Becomes read-only, visible to no one except the Founder. Preserves data for 1 year then auto-deletes.
- **Option C — League remains indefinitely as a ghost.** Never deleted.

**Recommendation: Option A**, with a confirmation prompt on leave: "You're the last member. Leaving will delete this league and all its data. Continue?" A one-person league with no other member has nothing to preserve. Option B bloats Firestore with dead data forever. Option C is worse.

**RESOLVED — 6.2.a.** Confirm Option A.

**>>> RESOLVED:** Option A confirmed. Auto-delete with explicit confirmation prompt. The rule applies literally at `memberUids.length === 1` (not "functionally inactive" or any softer heuristic). Founding league is exempt per the existing hard gate.

## 6.3 Founder (Zach) deletes own account

The existential scenario. Not likely, but must be answered.

**What should happen:**

- Account deletion is **gated**. Zach cannot delete his account via the normal flow.
- If he genuinely wants out, the flow is: "You are the platform Founder. Deleting this account is a platform-ending action. Contact [emergency email] or run [documented CLI script] to proceed."
- The founding league is untouched if Zach simply wants to retire — the Commissioner role of the founding league can be transferred first (to Jordyn, Nick, whoever), and then the Founder role can be formally dissolved (or not — the Founder role can simply go dormant).

**Implication for the founding league:** even if Zach-the-Founder is no longer, the founding league's "founding" badge is permanent. The league persists; only the Founder role is vacated.

**RESOLVED — 6.3.a.** If Zach formally retires as Founder, does the role transfer to someone else or become permanently vacant?

- **Option A — Transfer to a nominated successor.** Continuity of moderation.
- **Option B — Vacant. Platform-wide support falls to... whoever is willing.**
- **Option C — Platform becomes unmaintained. An "abandoned platform" banner appears.**

**Recommendation: Option A.** Platforms without a backstop decay. Successor should be nominated explicitly in a governance document (likely off-platform, e.g., a written note); Zach picks them.

**>>> RESOLVED:** Option A. The transfer mechanism exists for emergency/continuity purposes (death, incapacity, extended absence). Zach does not intend to step away. The identity of any potential successor is a private off-platform decision, not baked into governance code. The platform supports the mechanism; who and when are handled privately.

## 6.4 Commissioner removes a member who had rounds logged

The member gets removed from the league. Their rounds remain in Firestore — they're historical facts.

**Proposed behavior:**

- **Member's rounds remain visible** on historical league leaderboards, records, and season standings. They happened; they are counted.
- **Member's name renders as "Former Member"** or similar on those historical surfaces (parallel to how the current "removed" role handling works, but with a visible label rather than silent removal).
- **Member's profile in the member list is gone.** They don't appear in current roster, member directory, tee time RSVP, etc.
- **Member's DMs:** the threads remain; the removed member loses access (they can't post or read further) but the other party's history is preserved.
- **Member's ParCoins and achievements travel with the account** (they are global per CLAUDE.md), not per-league. If the member joins another league later, they bring those.

**RESOLVED — 6.4.a.** Should league season standings recalculate retroactively when a member is removed, erasing them from past standings?

- **Option A — No.** Historical fact. Standings are snapshots.
- **Option B — Yes.** The league is a fresh social contract; a removed member didn't deserve the points they earned.
- **Option C — Yes, but only for current-season standings.** Past seasons are frozen.

**Recommendation: Option A.** Retroactive erasure breaks trust in records. Current-season standings are important enough to be live data; past seasons are history books.

**>>> RESOLVED:** Option A. Historical standings frozen. Removed members appear on historical surfaces (past leaderboards, season archives, course records) with a "Former Member" label rather than silent removal — preserves the honest record while signaling the person is no longer present.

## 6.5 Banned user creates a new account

They signed up with a new email after being platform-banned.

**Enforcement layers, ordered by strength:**

- **Email match** — trivial to bypass (new email), but catches the lazy.
- **Device fingerprint** — harder to bypass, privacy-invasive to collect.
- **IP match** — network-address-based; bypassed by VPN or change of location, catches overlap.
- **Manual review** — Founder sees a new registration, recognizes the person, bans again.

**RESOLVED — 6.5.a.** How hard do we try to prevent re-entry?

- **Option A — Email block only.** Ban the email; no further enforcement. Simple, easily defeated.
- **Option B — Email + IP match flag for Founder review.** New registrations from a banned IP surface in an admin queue.
- **Option C — Full fingerprinting + IP.** Invasive, best coverage. Requires a fingerprinting library and a privacy policy update.

**Recommendation: Option A at 17-member scale; revisit at scale.** At 17 members, everyone knows who is who. At 1000+, consider Option B. Option C is a reach for a golf app.

**>>> RESOLVED:** Option A at current scale. Email block plus immediate invalidation of active auth sessions on ban. No IP matching, no fingerprinting. Revisit when the platform exceeds ~1,000 users or when repeat-offender patterns actually emerge.

## 6.6 Two commissioners of different leagues dispute shared data (H2H)

Example: Member A (League X) and Member B (League Y) are friends; they have head-to-head cross-league rounds. Commissioner X wants those rounds hidden from League X's records because he claims Member B is a sandbagger. Commissioner Y disagrees.

**Proposed behavior:**

- **No commissioner can edit another league's records.** Period. Per 3.5, only Founder has cross-league data-modification rights.
- **Each league decides for itself** what counts in its records. H2H cross-league rounds appear in both leagues' per-member record displays, but each league can independently flag or exclude.
- **If a commissioner wants to exclude a specific cross-league round from his league's records**, the feature is: "flag as unofficial for this league." The round still exists; it's not counted in Season X standings.
- **If the dispute is between members themselves**, the Founder is the tiebreaker of last resort — but in practice, members and commissioners should resolve it socially first.

**RESOLVED — 6.6.a.** Should "flag as unofficial for this league" be a feature in v8, or can it be deferred? Recommendation: defer. It's an escape valve we might never need; adding it now is premature optimization. Revisit if a real dispute arises.

**>>> RESOLVED:** Defer. Not in v8 scope. Cross-league round disputes are resolved socially or escalated to the Founder. Revisit only if real dispute frequency materializes.

## 6.7 Member appeals a ban or suspension

No appeal flow exists today. Proposal:

- **Suspended member** can see a banner on their (read-only) home: "Your account is suspended until [date]. Reason: [reason]. [Appeal]" button → opens a form to the Founder.
- **Banned member** can see a one-screen "Your account has been banned. [Appeal]" with a form to the Founder. The form is simple: name, email, reason for appeal, free text.
- **Appeals land in the admin panel's reports area** under a "Appeals" subsection. Founder reviews and responds via email or by lifting the ban.

**RESOLVED — 6.7.a.** Should appeals be time-limited (e.g., one appeal per ban, no resubmission for 30 days) to prevent spam? Recommendation: yes — one appeal, then a 30-day cooldown.

**>>> RESOLVED:** Yes. One appeal per 30-day cooldown.

## 6.8 Does the Founder role have "can't be removed" protection?

The founding league has ironclad protection. Parallel question for the Founder role:

**Proposed:**

- **Yes.** The Founder role cannot be revoked by any in-app action, including by the Founder themselves via the normal account flow. The only revocation path is the documented gated flow in 6.3.
- The rule: `isFounder(uid)` is a function of a hardcoded uid (or a one-row config doc written once at platform inception and then locked). It cannot be mutated by Firestore rules or by client code.
- **One exception:** an explicit transfer flow, documented at the code level, that requires both the current Founder's consent and the new Founder's acceptance, with a 7-day cooling-off window.

**RESOLVED — 6.8.a.** Approve the above "can't be removed except via explicit gated transfer" protection?

**>>> RESOLVED:** Yes, approved. `isFounder(uid)` checks against a platform-level config document set at inception and locked. No UI and no Firestore rule allows Founder revocation except through the gated transfer flow: initiate + accept + 7-day cooling-off, revocable during the cooling-off period.

## 6.9 Secondary edge cases (lower priority)

### 6.9.a — A commissioner is platform-suspended by the Founder

Per Section 2.6, platform suspension is separate from league role. A commissioner who is platform-suspended cannot act as commissioner until the suspension lifts — but their commissioner *role* is preserved (doesn't revert to "member"). When suspension lifts, they resume commissioner duties.

### 6.9.b — A commissioner is platform-banned by the Founder

Same as above, but permanent. The commissioner role transfers via force-transfer to the longest-tenured admin, or if none exists, the league is archived pending Founder decision.

### 6.9.c — League commissioner wants to leave their own league but not the platform

Same as 6.1 Scenario B: must transfer commissioner role first, then leave. The app should surface this constraint clearly ("You're running this league. Transfer commissioner to leave.").

### 6.9.d — Two users simultaneously try to accept a commissioner transfer

Concurrency edge case. Handled at the data-layer (Firestore transaction ensures only one write succeeds). From a UX standpoint, the losing user sees "This transfer has already been accepted by someone else." Rare enough to punt on special UI.

### 6.9.e — A league hits 0 admins, commissioner still present

Totally legal per 2.3.a Option A (no minimum). No intervention needed.

### 6.9.f — A member is in 50+ leagues

Per CLAUDE.md's Multi-League Architecture section, multi-league is unlimited at Phase 1–3. 50 leagues is legal. The leagues-list UI needs pagination or search when this gets absurd — but that's a UI polish concern, not a governance concern.

### 6.9.g — A public league hits a "viral" growth spike (100+ join requests in 24 hours)

Commissioner + admins are buried in approvals. **RESOLVED — 6.9.g.1.** Should the app offer an "auto-approve everyone meeting criteria X" option, or is that out of scope for v8.0? Recommendation: out of scope. v8 is governance foundations; auto-moderation is a v9 topic.

**>>> RESOLVED:** Out of scope for v8. When real demand materializes, the first response is batch actions via checkbox selection (select multiple requests → approve all / deny all), with an optional rule-based auto-approve toggle. Specifics deferred to v9.x.

## 7.1 Backfill strategy

### 7.1.a — Platform roles

# Section 8 — Open Questions

## 8.1 The "Commissioner" word

The word "Commissioner" currently refers to Zach-the-platform-operator. Under v8's proposal, it refers to any person running any league. Collision risk.

**RESOLVED — 8.1.a.** Rename one of them?

- **Option A — Keep both as "Commissioner"; disambiguate by scope (platform-wide "The Commissioner" vs. per-league "your league's Commissioner").** Clean conceptually but confusing in copy.
- **Option B — Rename the platform-wide role to "Founder."** Matches Section 2.1. "Commissioner" becomes a per-league word. This is the recommendation through this whole document, but needs explicit approval because a lot of copy (FAQ, rules, onboarding, merch) uses "The Commissioner" for Zach.
- **Option C — Keep "Commissioner" for Zach; rename per-league role to "League Owner" or "League Captain."** Preserves existing copy but invents a new word for a well-understood concept.

**Recommendation: Option B.** Founder is accurate (Zach founded the platform); Commissioner becomes a recognizable per-league role. Copy cleanup is a one-time cost.

**>>> RESOLVED:** Option B. Platform-wide role renamed from "Commissioner" to "Founder." Per-league role remains "Commissioner." All existing copy across FAQ, rules, onboarding, merch, and settings gets updated during v8.0.1 Visibility Polish to reflect: Founder = platform-level (Zach), Commissioner = per-league.

## 8.2 How does Parbaughs the platform brand interact with individual league brands?

When users say "I play on Parbaughs," do they mean Parbaughs-the-platform or their specific league? The Parbaughs-the-league (founding league) has a stronger brand today, which creates ambiguity.

**This is a branding/marketing question more than a governance one**, but it matters for role visibility. If leagues end up with unique identities (logos, themes, custom branding), the visibility matrix changes — the League Commissioner treatment should be league-themed, not platform-gold.

**RESOLVED — 8.2.a.** Is league-custom branding in scope for v8, or punted to v9+?

**>>> RESOLVED:** Out of scope for v8. When built, it is a paid commissioner feature. Monetization gates validate commitment before the platform invests in design, storage, and moderation overhead. It requires its own Stage 1 design doc covering: upload moderation, theme interaction with user cosmetics, share card behavior, storage economics, and feature pricing tier. Likely Phase 4 monetization scope. Stub parked at `docs/league-custom-branding-design.md`.

## 8.3 Should leagues have "private" moderation data (audit logs, moderation notes)?

A commissioner removes a member. Is there a private log only the commissioner and admins can see ("Member X removed on date Y by Z, reason: ...")? Or is removal silent?

**Proposed:** yes, a league-scoped audit log visible only to the league's commissioner and admins. Prevents "he said / she said" disputes.

**RESOLVED — 8.3.a.** In scope for v8, or v9? Recommendation: v9. It's a polish feature, not a foundation.

**>>> RESOLVED:** Minimal log in v8. All kick and ban actions write log entries: action type, target user, actor uid, timestamp, optional reason (with private-flag support). Visible to the league's commissioner and admins only; private-flag reasons are hidden from everyone except the Founder. Simple list UI on the league settings page — no search, filters, or export. Expanded audit logging (rename events, visibility changes, admin role changes, notifications, Founder rollup views) is deferred to v9.x. Shipping the minimal version in v8 prevents the "he said / she said" dispute class from landing in the first ban case and scales naturally to the fuller audit view later.

- If a user's subscription lapses and they're in 10 leagues, what happens to the leagues where they're commissioner?
- Does the Founder role grant access to paywalled features regardless of subscription? (Obviously yes for Zach, but what if the role transfers?)

These are **out of scope** for this document but must be answered before Phase 4 ships.

## 8.5 Legal and compliance

- **Minors:** CLAUDE.md does not specify an age policy. If a league has members under 13 (COPPA in US), there are legal implications for DMs, data storage, and moderation. Not a v8 concern today but will be before App Store launch if unaddressed.
- **Right to be forgotten (GDPR, CCPA):** account deletion today deletes Firestore member doc + photos + auth account (`firebase.js:458-495`). It does **not** delete rounds, chat history, DMs sent by the user, or ParCoin transactions. **RESOLVED — 8.5.a.** Is this compliant with the terms we want to offer? Recommendation: explicitly opt into a "full erasure" path for users who request it (anonymize their content rather than delete, so historical records remain coherent).

  **>>> RESOLVED:** Anonymization-based erasure flow. Two account-removal paths coexist:
  - **"Delete my account"** (existing): removes auth, member doc, and photos; rounds/chat/DMs remain attributed to the display name at time of deletion.
  - **"Erase me completely"** (new): anonymizes all user references (display name → "Former Member", username → `deleted_user_[id]`, email/bio/photo/location → null), deletes auth, preserves historical content under the placeholder.

  Every erasure creates an audit log entry for legal compliance. Implementation requires a Cloud Function running against Admin SDK. Must ship before App Store submission.
