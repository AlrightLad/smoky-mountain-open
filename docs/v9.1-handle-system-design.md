# v9.1 Handle System — Design Stub

**Status:** Parked. Awaiting Stage 1 design work.
**Created:** 2026-04-17
**Blocks:** v9.1, user search UX, friend system discovery (v9.0).

## Context

Today's Parbaughs uses `username` (letters, numbers, underscores, 3–20 chars, globally unique) as both identity and display key. This works at 17 members but breaks at scale: two people named "Zach" can't both have `zach` as a username, and search by display name is ambiguous ("there's a Zach in every crew").

v8.0 governance surfaced the problem via the friend system (v9.0) — users need to find each other by handle without requiring exact display name match. Discord's `username#discriminator` pattern solves this: your username doesn't have to be unique; the discriminator makes the full handle unique.

## High-level direction (from v8.0 governance review)

- Adopt a Discord-style `username#discriminator` format.
- Enables search and friend-add without requiring an exact display-name match.
- Handles are unique via the discriminator suffix — multiple users can share `zach` as their display name.
- Existing 20+ members need a migration path — can't break their current identities.

## Open design questions

- **Discriminator length.** 4 digits (`zach#1234`) gives 10,000 collisions per name. 6 digits gives 1M. At 1,000 users, 4 is plenty; at 100,000 with popular names, 4 starts to pinch. Pick one and hold.
- **Random vs user-chosen discriminator.** Random avoids the "I want #0001" gold-rush pattern and the moderation overhead of vanity handles. User-chosen is a monetization lever (Nitro-style paid vanity handles). Decide based on whether vanity handles should be a revenue stream.
- **Change-handle mechanics.** How often can a user change their handle? Free or paid? Does changing reassign the discriminator, or keep it?
- **Migration for existing members.** Each current user needs a discriminator assigned. Options: (a) autogenerate at migration time, (b) let users pick during a one-time migration prompt, (c) reserve `#0001`–`#0020` for the founding cohort. (c) aligns with "founding four" narrative but adds complexity.
- **Reserved handles.** Block platform-sensitive names: `admin`, `founder`, `commissioner`, `support`, `parbaughs`, plus common bad-word lists. Who maintains the blocklist?
- **Search behavior.** Exact match on `name#discriminator`. Fuzzy match on `name` alone (returns all collisions; user disambiguates). At what threshold does fuzzy match return too many results to be useful?
- **Legacy `username` field.** Keep for internal URL routing? Deprecate entirely? Display-name and handle can coexist (Discord has both), but the username in the current Firestore schema may need restructuring.
- **Display in UI.** Show full `name#discriminator` always, or only on disambiguation-sensitive surfaces (search, friend requests)? Discord shows handle in-profile and display name everywhere else.

## Not in scope for this doc

- Friend system mechanics — v9.0's concern. This doc defines how handles enable discovery; v9.0 defines what users do with that discovery.
- Paid vanity handle pricing (if that's a direction) — lives in Phase 4 monetization design.
- Implementation details — Firestore indexes, migration scripts, rules changes. All Stage 2.
