## 1.1 Current member doc schema (`members/{uid}`)

The member doc is the authoritative record for a user. Schema is not enforced in Firestore rules — any field, any value is writable by the owning user or commissioner (`firestore.rules:12`). Fields are written from multiple paths; the canonical write sites are in `src/core/firebase.js` during registration.

### Identity fields

- `id` (string) — the Firebase Auth UID for real accounts. For the local-fallback path (`firebase.js:542`) it is the literal string `"local"`. Set at registration (`firebase.js:316, 355`).
- `email` (string) — auth email. Set at registration (`firebase.js:318, 355`).
- `username` (string, lowercased) — unique handle. Enforced via pre-registration query (`firebase.js:278`). 3–20 characters, regex `^[a-zA-Z0-9_]+$` (`firebase.js:252`).
- `name` (string) — display name. Often equals `username` at registration but is independently editable later.
- `nick` (string) — optional display suffix shown below the name. Member-editable (`members.js:1172`).

### Authorization fields

- `role` (string, one of `"commissioner"` | `"member"` | `"suspended"` | `"removed"`) — platform-level authorization. Values:
  - `"commissioner"` — assigned once to the first FOUNDING-FOUR registrant (`firebase.js:300-302`). All platform-level capabilities gated on this value.
  - `"member"` — default for every other user (`firebase.js:358`, `firebase.js:503`, `firebase.js:534`).
  - `"suspended"` — set by admin panel (`admin.js:328`).
  - `"removed"` — set by admin panel (`admin.js:372`).
- `claimedFrom` (string, optional) — for founding-four members only. Contains the pre-migration seed id (e.g. `"testzach"`). Used by `PB.getAllPlayerIds` (`data.js:157-172`) to merge historical rounds. Set at registration only for FOUNDING-FOUR code (`firebase.js:317, 304-307`).
- `isFoundingFour` (boolean) — true only when user registered with FOUNDING-FOUR code (`firebase.js:339`).
- `founding` (boolean) — duplicate of `isFoundingFour`, used by display paths (`members.js:115, 139, 190, 300, 450`).
- `badges` (array of strings) — seeded with `["founder"]` for FOUNDING-FOUR users (`firebase.js:341`). Never rendered visually (flagged in Section 1.8).

### Suspension / removal companion fields

- `suspendedAt` (timestamp) — when role flipped to suspended (`admin.js:329`).
- `suspendedUntil` (timestamp) — intended end of suspension (`admin.js:330`). **Not enforced.**
- `suspendedBy` (string uid) — who issued the suspension (`admin.js:331`).
- `suspendedReason` (string) — optional explanation (`admin.js:332`).
- `removedAt` (timestamp) — when role flipped to removed (`admin.js:373`).
- `removedBy` (string uid) — who issued the removal (`admin.js:374`).
- `removedReason` (string, optional) — explanation (`admin.js:375`).

### League membership fields

- `leagues` (array of string leagueIds) — every league this user belongs to. Written at registration (`firebase.js:346, 362`), modified on join (`leagues.js:195`), leave via delete (`leagues.js:474`), approval (`leagues.js:396`).
- `activeLeague` (string leagueId) — currently selected league. Written at registration, on switch (`leagues.js:286`), on join (`leagues.js:196`), on league delete if active (`leagues.js:484`).

### Stats and progression

- `xp` (integer) — global experience points, computed from rounds by `PB.getPlayerXP`. Persisted by `persistPlayerStats` (`sync.js:127`).
- `level` (integer) — derived from xp.
- `totalRounds` (integer) — global round count, persisted alongside xp.
- `avgScore` (integer or null) — mean of 18-hole public, non-scramble rounds.
- `bestRound` (integer or null) — minimum of same set.
- `computedHandicap` (number or null) — from `calculateHandicapIndex`.
- `handicap` (number or null) — legacy field, mirrors `computedHandicap`. Kept for backward compatibility (`fixtures/users.js:181-182`).
- `manualHandicap` (number or null) — user-entered handicap for those who don't have enough rounds.
- `wins` (integer) — career wins count.
- `trips` (integer) — trip participation count.
- `updatedAt` (timestamp) — last time `persistPlayerStats` wrote. Refreshed on every persist (`sync.js:166`).

### ParCoin economy

- `parcoins` (integer) — current balance (`parcoins.js:56, 101`).
- `parcoinsLifetime` (integer) — lifetime earned, for Gold Member status (`parcoins.js:58`).
- `activeDoubleXP` (boolean) — power-up flag (`richlist.js:108`).
- `handicapShield` (boolean) — power-up flag (`richlist.js:112`).
- `sponsoredHole` (object `{course, season}`) — sponsor flex item (`richlist.js:133`).
- `namedTournament` (string) — name-a-tournament purchase (`richlist.js:142`).

### Cosmetics and equipped items

- `theme` (string, theme id) — user-selected theme (`theme.js:44`).
- `title` (string) — unlocked title history.
- `equippedTitle` (string) — currently-shown title (`members.js:1837`).
- `displayBadges` (array of badge ids) — badges to render (`members.js:1867`).
- `ringStyle`, `banner`, `nameEffect`, `cardTheme` (strings) — equipped shop items.
- `photoUrl` (string, optional) — avatar. Stored in `photos` collection (`firebase.js:712-731`), URL cached here for quick lookup.

### Activity and preference

- `onboardingComplete` (boolean) — true once user finishes onboarding (`onboarding.js:100`).
- `profilePublic` (boolean) — public-profile toggle (`settings.js:181`).
- `bio`, `homeCourse`, `favoriteCourse` (strings) — profile fields (`members.js:1394`).
- `clubs`, `bag`, `funnyFacts`, `range`, `bagPhoto` — profile extras (`firebase.js:328-331`).
- `emoji` (string) — member emoji shown in some places.

### Invite quota

- `maxInvites` (integer) — invites available. `999` for commissioner, `3` default (`firebase.js:336, 360`).
- `invitesUsed` (integer) — increments on successful referral (`invite.js:70`, `members.js:1724`).
- `invitedBy` (string) — uid of the inviter (`firebase.js:338, 359`).

### Sharing and analytics

- `shareCount` (integer) — total share-card generations.
- `sharedRounds` (array of round ids) — rounds the user has shared (`firebase.js:506-507`).

### Notification / push

- `fcmToken` (string, optional) — FCM registration token (`router.js:395`; `functions/index.js:59`).
- `fcmUpdatedAt` (timestamp) — token refresh time.

### Auto-maintained

- `createdAt` (timestamp) — set once at registration (`firebase.js:347, 363`).
- `earnedAchievements` (array of achievement ids) — computed list (`router.js:1521, 1524`).

### Field type summary

The doc is effectively a grab-bag of typed fields. There is no schema validation in the client or in rules. Partial writes via `.update()` (not `.set({merge:true})`) are the norm. This matters for v8: any new field we add needs an explicit population step during migration (Section 6) or needs to be treated as optional for all users who predate it.

## 1.2 Current league doc schema (`leagues/{leagueId}`)

Canonical write at `src/pages/leagues.js:123-140`. Updates scattered across `leagues.js` (visibility toggle, admin toggle, invite regeneration).

- `name` (string) — display name, 3+ chars (`leagues.js:117`).
- `slug` (string) — used as the doc id. Derived from name (`leagues.js:119`).
- `location` (string) — free text.
- `description` (string) — free text.

- `badge` (string) — `"founding"` for the-parbaughs, empty otherwise. Hard-gated as non-deletable (`leagues.js:258, 470`).

- `admins` (array of uids) — per-league admin list (`leagues.js:133`). Seeded with `[creator.uid]`.
- `memberCount` (integer) — convenience counter, maintained with `FieldValue.increment` (`leagues.js:192, 393`).
- `memberUids` (array of uids) — full roster (`leagues.js:135`). Maintained with `FieldValue.arrayUnion` (`leagues.js:191, 392`) and `arrayRemove` (on league delete, `leagues.js:474`).
- `inviteCode` (string, format `LG-XXXXXXXX`) — league-level invite (`leagues.js:120`, regen at `leagues.js:453`).
- `theme` (string) — default `"classic"`. Currently unused in UI for per-league theming (see Parked Design Tracks — league custom branding).
- `requireApproval` (boolean) — gating for join requests (`leagues.js:441`).
- `createdAt` (timestamp).
- `settings` (object) — `{ seasons, parcoins, wagers, bounties, trashTalk }` flags. Not all surfaces consult these yet.

### Subcollection

- `leagues/{leagueId}/joinRequests/{uid}` — per-join-request docs with `{ uid, name, handicap, level, homeCourse, createdAt, status }`. Status flips `"pending"` → `"approved"` | `"denied"` (`leagues.js:324, 399, 410`). No security rules for this subcollection in the current `firestore.rules` — read/write relies on the parent rule or catch-all, which **isn't present** per the `// NO CATCH-ALL` comment at `firestore.rules:141`. In practice Firestore denies all access by default. **This means join requests only work in production today if the rules file has been patched outside the repo, or requests silently fail.** Flagged as a risk in Section 10.

## 1.4 Current auth flow — deep dive

Because Section 4 is the auth-flow priority area, this subsection establishes a thorough baseline of what happens today from sign-in through Router.go("home"). Cite every line.

### 1.4.a — Auth state subscription

`src/core/firebase.js:498-538`:

```js
if (firebaseAvailable && auth) {
  auth.onAuthStateChanged(function(user) {
    currentUser = user;
    if (user) {
      db.collection("members").doc(user.uid).get().then(function(doc) {
        currentProfile = doc.exists ? doc.data() : { id:user.uid, name:user.displayName||"Member", role:"member" };
        window._pbShareCount = (currentProfile.shareCount || 0);
        // Apply theme from Firestore profile
        if (currentProfile.theme && THEMES[currentProfile.theme]) {
          applyTheme(currentProfile.theme);
          try { localStorage.setItem("pb_theme", currentProfile.theme); } catch(e) {}
        }
        enterApp();
        // Start real-time profile listener
        if (window._memberProfileUnsub) window._memberProfileUnsub();
        window._memberProfileUnsub = db.collection("members").doc(user.uid).onSnapshot(function(snap) {
          if (!snap.exists) return;
          currentProfile = snap.data();
          // ... theme sync ...
          updateProfileBar();
          var pg = Router.getPage();
          if (pg === "home" || pg === "members" || pg === "settings") {
            Router.go(pg, Router.getParams(), true);
          }
        });
      }).catch(function() {
        currentProfile = { id:user.uid, name:user.displayName||"Member", role:"member" };
        enterApp();
      });
    } else { exitApp(); }
  });
}
```

Key observations for v8:

1. **Single profile doc read at auth time** (line 502) returns the member doc or a fallback synthetic profile. The fallback always assigns `role: "member"` — there's no notion of "this user exists in Auth but not in Firestore yet." In v8.0 we need to handle a third case: the user has a member doc but `platformRole: "banned"` — see Section 4.
2. **Theme is applied before `enterApp`** (line 510). `enterApp` fires synchronously after the Firestore read resolves.
3. **`_memberProfileUnsub` is registered AFTER enterApp** (line 517). This means the home page renders initially from the synchronous Firestore read, then re-renders when the snapshot listener fires with its initial delivery. At emulator speed this is imperceptible; on slow connections it's noticeable.
4. **The listener replaces `currentProfile` on every doc change** (line 519). Any client-initiated field update to the member doc round-trips and replaces the entire profile. This is load-bearing for how v7.9's session-start `persistPlayerStats` converges UI state after its write.
5. **Auth signout (`else { exitApp(); }`)** at line 537 just tears down UI — it doesn't clear `currentUser` or `currentProfile` explicitly. Those get overwritten on next sign-in but remain stale in between. This matters for mid-session suspension/ban: we need explicit cleanup.
6. **No `platformRole` handling anywhere.** Every check on the profile uses `currentProfile.role`.

### 1.4.d — Session invalidation

There's no concept of server-side session invalidation in the current code. When admin.js sets `role: "suspended"` or `role: "removed"` (`admin.js:327, 371`), the active session on the target user's device continues indefinitely. The `_memberProfileUnsub` listener fires and the value of `currentProfile.role` updates to `"suspended"`, but no code path reads that value to enforce anything. The member card display (`members.js:160`) merely shows a red "Suspended" label.

This is the single largest enforcement gap in current state. Section 4 addresses it.

## 1.8 Known enforcement gaps

Confirmed during this audit and in the Stage 1 governance recon:

1. **Suspension is cosmetic.** No Firestore rule references `role == "suspended"`. A suspended user can still write to their own member doc, log rounds, post chat, DM, etc. The admin panel UI + the member card display are the only visible evidence of the state. (`admin.js:245, 249, 270`; `members.js:160`.)

2. **Removal is cosmetic.** Same issue. A removed user is hidden from rosters client-side (seven filter sites listed in 1.5.b) but has full server-side access. (`data.js:140`, `findplayers.js:33`, `dms.js:24`, `partygames.js:79`, `rounds.js:315`, `members.js:61`, `teetimes.js:177`.)

3. **League delete is broken.** `confirmDeleteLeague` (`leagues.js:462-491`) calls `db.collection("leagues").doc(lid).delete()`. `firestore.rules:71` says `allow delete: if false`. The client flow fails silently because the `.delete()` promise is not awaited (`leagues.js:478-489` chains a `.then` but the outer `.catch` is missing). In v8, either ship a commissioner-delete rule + cascade function, or remove the dead UI.

4. **Bounties update rule is overly permissive.** `firestore.rules:86` — `allow update: if isAuth();`. Any authenticated user can modify any bounty. Should be scoped to bounty owner or claimant.

5. **Trips and tripscores are unrestricted.** `firestore.rules:121-122` — `allow write: if isAuth();` for both. Any user can modify any trip or its scores. This has been live since v6.x and hasn't caused issues because the tight 17-member user base doesn't abuse it. At scale it's a problem.

6. **rangeSessions allow delete by any auth user.** `firestore.rules:131` — `allow update, delete: if isAuth();`. A malicious user could wipe other users' range sessions. Low-likelihood, high-impact.

7. **parcoin_transactions is create-by-anyone.** `firestore.rules:59` — `allow create: if isAuth();`. The comment says "immutable transaction log" but the creation is unrestricted — any authenticated user can forge transactions with someone else's uid in the payload. The read side filters by `resource.data.uid == uid()` so forged transactions wouldn't be visible to the victim, but they'd still inflate Gold Member leaderboards or other aggregate reads that don't filter.

8. **Missing join-request rules.** `leagues/{leagueId}/joinRequests/{uid}` is written at `leagues.js:316` but has no rule in `firestore.rules`. The system only works because the production rules file may have been amended outside the repo, or because join requests silently fail.

9. **No platform-level session kill.** There's no mechanism to revoke an active auth session. Suspending or banning a user in the admin panel has zero immediate effect on their logged-in device.

10. **`role` field overloaded.** Already covered in governance doc; restating here for completeness. Setting `role: "suspended"` overwrites a commissioner's role permanently — the reinstate path sets `role: "member"` (`admin.js:347`), not back to commissioner.

These all get addressed in Section 3 (rules rewrite) and Section 4 (auth flow).

# Section 2 — Target State Schema

Every collection that changes under v8.x, with before and after. Format:

- **BEFORE:** what's in v7.9.1 today.
- **AFTER:** what v8.0 lands with.
- **CHANGE REASON:** which governance decision drives it.
- **MIGRATION NOTE:** how existing data becomes compliant.

Fields not mentioned in "AFTER" are unchanged.

## 2.1 `members/{uid}` — modified

### BEFORE

Full schema enumerated in Section 1.1. The authoritative platform-level role is `role` (values `"commissioner"` | `"member"` | `"suspended"` | `"removed"`).

### AFTER

**New fields:**

- `platformRole` (string, one of `"founder"` | `"user"` | `"suspended"` | `"banned"`). Replaces the platform-level meaning of `role`.
- `suspension` (object, optional): `{ until (timestamp), reason (string), reasonPrivate (boolean), issuedBy (uid), issuedAt (timestamp) }`. Present only when `platformRole == "suspended"`.
- `ban` (object, optional): `{ reason (string), reasonPrivate (boolean), issuedBy (uid), issuedAt (timestamp) }`. Present only when `platformRole == "banned"`.
- `pendingErasure` (boolean, optional) — flagged when the user initiates "Erase me completely" but the Cloud Function hasn't completed yet. Triggers read-only gating during the erasure window.

**Removed fields (eventually):**

- `role` — in v8.0.0 it's a **dead-read field** (kept for backward compatibility). Display sites still read it as a fallback. In v8.1 every display site is migrated to read `platformRole`. In v8.2.0 the field is deleted from every doc and the rule references purged. See decision 7.1.c.1.
- Legacy `suspendedAt`, `suspendedUntil`, `suspendedBy`, `suspendedReason` — absorbed into `suspension`. Kept until v8.1.0 migration completes.
- Legacy `removedAt`, `removedBy`, `removedReason` — absorbed into `ban`. Same timeline.

**CHANGE REASON:**
- Decision 2.1.a: Founder stored as explicit `platformRole` field, not uid equality.
- Decision 2.6.a: Split `role` into `platformRole` + per-league role tables. Platform states (suspended, banned) belong on `platformRole`; per-league roles live on league doc.
- Decision 3.2.a: Ban reason private-flag support — applied to both platform `ban.reasonPrivate` and league ban records.

**MIGRATION NOTE:** Section 6.1 covers the exact backfill. Summary: Zach → `platformRole: "founder"`, every other current member → `platformRole: "user"`. Any `role: "suspended"` → `platformRole: "suspended"` with suspension object from legacy fields. Any `role: "removed"` → `platformRole: "banned"` with ban object from legacy fields.

## 2.2 `leagues/{leagueId}` — mostly unchanged

### BEFORE

Schema in Section 1.2. `commissioner` singleton, `admins` array, `memberUids`, `memberCount`, `requireApproval`, etc.

### AFTER

**New fields:**

- `pendingCommissionerTransfer` (object, optional): `{ to (uid), initiatedAt (timestamp), expiresAt (timestamp), coolingOffEndsAt (timestamp) }`. Present only when a commissioner transfer is in flight.
- `bans` (array of uids) — users banned from rejoining this league. Populated by per-league ban action (decision 3.2.a). Distinct from platform bans (which live on the member doc). Kept as array for fast `in` lookups in rules.

**Unchanged semantics, new enforcement:**

**CHANGE REASON:**
- Decision 2.2.a: Commissioner transfer handshake needs a doc field to track in-flight transfers.

**MIGRATION NOTE:** New fields are optional; legacy docs work unchanged. No backfill needed.

## 2.3 `platformConfig/{singleton}` — NEW

Single-document collection, doc id literal `"singleton"` (or `"v8"` if we prefer to version). Holds platform-level config that must be inspectable in rules and mutable-once-locked after initial write.

### Fields

- `founderUid` (string uid) — Zach's uid. Written once during v8.0.0 migration. Used by `isFounder(uid)` helper in rules (Section 3.2).
- `founderTransfer` (object, optional): `{ from (uid), to (uid), initiatedAt (timestamp), acceptedAt (timestamp), coolingOffEndsAt (timestamp), status ("initiated" | "accepted" | "cancelled" | "completed") }`. Present during the 6.3.a / 6.8.a founder transfer flow.
- `platformVersion` (string) — the app version that last ran migration. Lets future migrations detect which state they're starting from.
- `lockedAt` (timestamp) — when the doc was last locked for modification (on completion of a successful transfer, for instance).

**Rules:** read=auth (any auth user can discover who the Founder is — that's appropriate, it's public knowledge), write=only via Cloud Function with Admin SDK. Client-side writes are denied (`allow write: if false;`).

**CHANGE REASON:**
- Decision 2.1.a: Founder stored as explicit field, not hardcoded uid.
- Decision 6.3.a + 6.8.a: Founder transfer needs a locked-write location that survives UI manipulation.

**DESIGN NEEDED — 2.3.1.** What's the doc id? Options: `"singleton"` (clean, implies one-ness), `"v8"` (version-scoped, future-proof for schema evolution), `"main"` (generic). Recommendation: `"singleton"`. If a v9 config lands with different shape, add a second doc `"v9"` rather than evolving one doc. Low-stakes decision but worth confirming.

## 2.4 `founder_access_logs/{logId}` — NEW

Audit log written every time the Founder reads private league data. Backs decision 3.5.a Option C.

### Fields

- `founderUid` (string) — who performed the read. Always Zach unless founder transferred.
- `leagueId` (string) — which league's data was accessed.
- `dataType` (string, one of `"dm"` | `"private_chat"` | `"moderation_record"` | `"ban_reason"` | `"suspension_reason"`) — what kind of private content was read.
- `reason` (string, optional) — free text the Founder provides at access time ("supporting ticket #12 from commissioner X"). Surfaces in the audit UI visible to league members.
- `timestamp` (timestamp) — when the read occurred.
- `contextRefs` (array, optional) — pointers to specific docs read (e.g. `["dms/xy_123", "chat/m_456"]`). For fine-grained audit but with privacy consideration: we do **not** log the *content* of what was read.

**Rules:**
- create=Cloud Function only (via Admin SDK; can't be forged from client)
- read=members of the affected league (`leagueId` in their `leagues[]` array)
- no update, no delete (append-only log)

**CHANGE REASON:** Decision 3.5.a — Founder reads are logged and visible to affected users.

**DESIGN NEEDED — 2.4.1.** Do aggregate reads (e.g., the Founder pulls a list of all members in League X to help with a support ticket) generate a log entry? Governance 3.5.a says "Only private content reads log" and "Aggregate stats and public content reads don't log." But the boundary between "private content" and "aggregate" is fuzzy — for example, pulling all ban records for a league is *technically* aggregate but it's moderation-sensitive. Recommendation: if the data type is in the `dataType` enum above (which is a short, explicit list of flagged categories), log it. Everything else doesn't.

**DESIGN NEEDED — 2.4.2.** How is the `reason` field captured? Options: (a) a mandatory text input in the Founder admin UI before the read fires, (b) automated based on entry point (e.g., "opened from support ticket viewer"), (c) optional — Founder writes whatever. Recommendation: (a) for private-content reads specifically. Friction is the point — if Zach has to type a reason, the log stays honest.

## 2.5 `leagues/{leagueId}/moderation_log/{logId}` — NEW subcollection

Per-league moderation audit. Backs decision 8.3.a (minimal log in v8).

### Fields

- `actionType` (string, one of `"kick"` | `"ban"` | `"unban"` | `"promote_admin"` | `"demote_admin"` | `"transfer_commissioner_initiated"` | `"transfer_commissioner_accepted"` | `"transfer_commissioner_cancelled"` | `"force_transfer_by_founder"`). v8.0 ships with `kick`, `ban`, `unban` only. Others deferred to v9.x (governance 8.3.a).
- `targetUid` (string) — the user affected.
- `actorUid` (string) — who performed the action.
- `timestamp` (timestamp).
- `reason` (string, optional).
- `reasonPrivate` (boolean) — if true, reason visible only to the actor and Founder; other league leadership see action but no reason.
- `ttl` (timestamp, optional) — for time-limited kicks or temporary actions. v8.0 bans are permanent until unbanned, so this is unused at launch.

**Rules:**
- create=league commissioner or admin (scoped to own league), or Cloud Function
- read=league commissioner + admins (full access); Founder (full access); target user (only the action + timestamp, not the reason); **nothing** to other league members
- no update, no delete (append-only; commissioner error requires creating a counter-entry)

**CHANGE REASON:**
- Decision 8.3.a: Minimal moderation log in v8. Expanded views (rename events, settings changes) deferred to v9.x.
- Decision 3.2.a: Ban records have `reasonPrivate` flag.

**DESIGN NEEDED — 2.5.1.** Does the target user see the log entry at all, or does only the public-facing "you were banned" notification land with them? Governance 3.2.a says the ban record is visible to league leadership and, via private flag, hidden from others except Founder. Doesn't say whether the target user sees their own record. Recommendation: target user sees action type + timestamp + non-private reason via a personal "moderation history" view; private reasons hidden. Same pattern as their personal notification history.

**DESIGN NEEDED — 2.6.1.** How does the notification bell render platform announcements differently from per-user notifications? - `retainedDataSummary` (string) — what was preserved (rounds, chat — under the "Former Member" placeholder).

- read=user (self) + Founder
- update=Cloud Function only (state transitions)
- no delete (legal record)

**CHANGE REASON:** Decision 8.5.a — anonymization-based erasure flow with audit trail.

**DESIGN NEEDED — 2.7.1.** Does the subcollection survive on the member doc after the member has been erased? Yes — that's the point. The parent member doc is anonymized to placeholder values but NOT deleted, and the erasure_requests subcollection provides the legal audit trail. Confirming explicitly because it's counterintuitive.

**DESIGN NEEDED — 2.7.2.** How long is the "pending" → "processing" window? If a user requests erasure and we take 3 days to process it, their data is still live during those 3 days. Recommendation: Cloud Function fires within 60 seconds of the request being written. Set an alert if `status == "pending"` for more than 1 hour.

## 2.8 `transfer_requests/{requestId}` — NEW

Unified collection for both commissioner transfers (decision 2.2.a) and founder transfers (decisions 6.3.a, 6.8.a). Using one collection simplifies rules and UI.

### Fields

- `type` (string, one of `"commissioner"` | `"founder"`) — discriminator.
- `leagueId` (string, optional) — present when `type == "commissioner"`, identifies the league.
- `fromUid` (string) — who initiated the transfer.
- `toUid` (string) — the nominee.
- `initiatedAt` (timestamp) — when `fromUid` nominated.
- `acceptedAt` (timestamp, optional) — when `toUid` accepted. Null while pending.
- `expiresAt` (timestamp) — auto-cancel time if not accepted.
- `coolingOffEndsAt` (timestamp, optional) — for commissioner transfers per refinement to decision 2.2.a (24–48h cooling-off); for founder transfers per decision 6.8.a (7-day cooling-off).
- `status` (string, one of `"pending"` | `"accepted"` | `"cancelled"` | `"completed"` | `"expired"`).
- `cancelledBy` (uid, optional) — populated when `fromUid` revokes during cooling-off.
- `reason` (string, optional) — rationale the initiator provided. Shown to the nominee.

**Rules:**
- create=`fromUid` only, with validation that they hold the role being transferred
- read=fromUid, toUid, league admins (for commissioner transfers), Founder (always)
- update=toUid (accept), fromUid (cancel during cooling-off), Cloud Function (expire/complete)
- no delete (historical record)

**CHANGE REASON:** Decisions 2.2.a, 2.2.b, 6.3.a, 6.8.a — unified mechanism for role transfers with handshake + cooling-off.

**DESIGN NEEDED — 2.8.1.** What's the TTL on a transfer_request before it auto-cancels if unaccepted? For commissioner transfers, 7 days feels right (enough time for the nominee to consider). For founder transfers, 14 days (major decision, deserves more thought time). Recommendation: `expiresAt = initiatedAt + 7d` for commissioner, `initiatedAt + 14d` for founder.

**DESIGN NEEDED — 2.8.2.** During the cooling-off window (after accept, before completion), who holds the role? Two readings of decision 2.2.a:
- (a) The incoming commissioner is "pending" and the outgoing commissioner retains authority until cooling-off ends. This prevents the nominee from making drastic changes that the originator didn't anticipate.
- (b) The incoming commissioner holds authority immediately on accept, but the outgoing one can revoke the transfer until cooling-off ends.
Recommendation: (a). Safer. The outgoing commissioner retains the power they had when they initiated the transfer; cooling-off exists to undo mistakes, not to hand over the keys and hope the handoff sticks.

**DESIGN NEEDED — 2.8.3.** Can there be multiple pending transfers on the same league (old one not yet cancelled, new one initiated)? Recommendation: no. Rule: `create` denied if any transfer_request for this leagueId is `status == "pending" || "accepted"` without terminal state. Forces initiator to cancel the old one first.

# Section 3 — Firestore Rules Rewrite

**Primary deep-investigation section.** The current rules file (`firestore.rules`, 143 lines) is built around a single `isCommissioner()` helper that reads `member.role == "commissioner"`. Under v8 this rewrites to a function hierarchy with ~7 helpers, full platform-role enforcement, per-league role checks, suspended/banned blocking, and Founder read-any with audit triggers.

## 3.1 Current rules audit

Full rule-by-rule table is in Section 1.3. Highlights relevant to v8:

- **Every rule today either uses `isAuth()` or the single `isCommissioner()` helper.** No per-league authorization exists in rules.
- **Five rules are overly permissive** (see Section 1.8: bounties, trips, tripscores, rangeSessions updates, parcoin_transactions create).
- **Three rules are missing** — no explicit rules for the `leagues/{leagueId}/joinRequests/{uid}` subcollection, no rules for any v8 new collections.
- **Zero rules enforce suspended/banned state.** Both are client-side veils today.

## 3.2 v8.0 rules architecture — helper hierarchy

Proposed helpers, in dependency order (each depends only on earlier ones plus primitives). Every helper below assumes `request.auth != null` unless noted; `isAuth()` is the root.

### `isAuth()`

```
function isAuth() {
  return request.auth != null;
}
```

Unchanged from today.

### `uid()`

```
function uid() {
  return request.auth.uid;
}
```

Unchanged from today.

### `memberDoc(memberUid)`

```
function memberDoc(memberUid) {
  return get(/databases/$(database)/documents/members/$(memberUid)).data;
}
```

Helper to avoid repeating `get` paths. Every rule that checks another user's state uses this.

**Performance:** each `get()` inside a rule counts as a read against the rules quota. Rules are evaluated on every operation, so heavy rules can become a bottleneck. Firestore caches `get` results within a single rule evaluation, so `memberDoc(uid())` called twice in one rule evaluates once. Across evaluations (different operations), each invocation re-fetches. We need to be sparing.

### `myProfile()`

```
function myProfile() {
  return memberDoc(uid());
}
```

Convenience — most rules check the caller's own state.

### `platformConfig()`

```
function platformConfig() {
  return get(/databases/$(database)/documents/platformConfig/singleton).data;
}
```

Single-document read. Used by `isFounder`.

### `isFounder(memberUid)`

```
function isFounder(memberUid) {
  return isAuth() && platformConfig().founderUid == memberUid;
}

function amIFounder() {
  return isFounder(uid());
}
```

Reads the platformConfig singleton and compares. Cheaper than reading the member doc for every check.

**Performance concern:** every Founder check requires a get of `platformConfig/singleton`. For high-traffic rules (rounds, chat), this adds a constant-cost read. Acceptable tradeoff because the alternative — storing founder uid on every user's member doc — wastes storage and makes transfer harder.

### `platformRoleOf(memberUid)`

```
function platformRoleOf(memberUid) {
  return memberDoc(memberUid).platformRole;
}

function myPlatformRole() {
  return platformRoleOf(uid());
}
```

Returns the platform role string. Used by suspended/banned checks.

### `isPlatformSuspended(memberUid)`

```
function isPlatformSuspended(memberUid) {
  return isAuth() && platformRoleOf(memberUid) == "suspended";
}
```

### `isPlatformBanned(memberUid)`

```
function isPlatformBanned(memberUid) {
  return isAuth() && platformRoleOf(memberUid) == "banned";
}
```

### `amIActive()`

```
function amIActive() {
  return isAuth()
    && myPlatformRole() != "banned"
    && myPlatformRole() != "suspended";
}
```

The "can I do anything that isn't read-only?" gate. Used at the top of most write rules.

### `amIReadable()`

```
function amIReadable() {
  return isAuth() && myPlatformRole() != "banned";
}
```

Suspended users retain read access per decision 3.1.a; banned users don't.

### `leagueDoc(leagueId)`

```
function leagueDoc(leagueId) {
  return get(/databases/$(database)/documents/leagues/$(leagueId)).data;
}
```

### `isLeagueCommissioner(leagueId, memberUid)`

```
function isLeagueCommissioner(leagueId, memberUid) {
  return isAuth() && leagueDoc(leagueId).commissioner == memberUid;
}

function amILeagueCommissioner(leagueId) {
  return isLeagueCommissioner(leagueId, uid());
}
```

### `isLeagueAdmin(leagueId, memberUid)`

```
function isLeagueAdmin(leagueId, memberUid) {
  return isAuth() && memberUid in leagueDoc(leagueId).admins;
}

function amILeagueAdmin(leagueId) {
  return isLeagueAdmin(leagueId, uid());
}
```

### `isLeagueMember(leagueId, memberUid)`

```
function isLeagueMember(leagueId, memberUid) {
  return isAuth() && memberUid in leagueDoc(leagueId).memberUids;
}

function amILeagueMember(leagueId) {
  return isLeagueMember(leagueId, uid());
}
```

### `amILeagueLeadership(leagueId)`

```
function amILeagueLeadership(leagueId) {
  return amILeagueCommissioner(leagueId) || amILeagueAdmin(leagueId);
}
```

### `isBannedFromLeague(leagueId, memberUid)`

```
function isBannedFromLeague(leagueId, memberUid) {
  return memberUid in leagueDoc(leagueId).bans;
}
```

### Helper-usage patterns

For a league-scoped write action (e.g., post a chat message in league X):

```
allow create: if isAuth()
            && amIActive()
            && amILeagueMember(resource.data.leagueId)
            && !isBannedFromLeague(resource.data.leagueId, uid());
```

This composes four helpers. Each resolves to at most one `get()` call per rule evaluation, cached within the evaluation.

For Founder-only capabilities:

```
allow write: if amIFounder();
```

Single rule, one get to `platformConfig`.

## 3.7 Known rule edge cases

### 3.7.a — User suspended mid-rename

Scenario: User is writing a name change (`update` on their member doc). Mid-request, another session suspends them. The update arrives at Firestore with a `platformRole == "user"` assumption, but the actual state has flipped to `"suspended"`.

Resolution: rules evaluate against the current committed state. The write will be blocked because `amIActive()` returns false. The client sees a permission error. Acceptable — race is rare and fails safely.

### 3.7.b — Founder transfer in progress — who holds authority?

Per DESIGN NEEDED 2.8.2 recommendation: outgoing commissioner/founder retains authority until cooling-off ends. So `amIFounder()` still returns true for the outgoing founder during the cooling-off window. The new founder's `platformRole` is still `"user"` until the transfer completes.

Implementation: `platformConfig.founderUid` is only updated when `transfer_requests/{id}.status` transitions to `"completed"` — which only happens after `coolingOffEndsAt` has passed AND no cancellation fired.

### 3.7.c — User banned from league A but commissioner of league B

Platform role: `"user"`. League A bans: `uid in leagueA.bans`. League B commissioner: `leagueB.commissioner == uid`.

Rules evaluated:
- In League A: `isBannedFromLeague(A, uid())` returns true → write blocked.
- In League B: normal commissioner writes pass.
- Global actions (create a new league): `amIActive()` is true → allowed.

This is correct per governance (per-league bans are per-league).

### 3.7.d — Platform-suspended user tries to unlock their own suspension

Attack vector: suspended user directly writes `platformRole: "user"` to their own member doc.

Rule: `members/{memberId}` update requires `request.resource.data.platformRole == resource.data.platformRole`. So the write is denied — they can't modify that field.

Who CAN modify it? The Cloud Function triggered by the admin panel's reinstate action. That function uses Admin SDK which bypasses rules.

### 3.7.e — Cascading league deletion

When a commissioner deletes their sole-member league, the rule allows the delete. But subcollections (`joinRequests`, `moderation_log`) remain. Firestore doesn't automatically cascade-delete subcollections.

Solution: a Cloud Function `onDelete(leagues/{leagueId})` sweeps subcollections. See Section 5.

### 3.7.f — New member joining an existing league

`leagues/{leagueId}` update rule requires `amILeagueCommissioner(leagueId) || amIFounder()`. But when a user joins, they write their own uid to `memberUids` — and they're not yet a member.

Current code path (`leagues.js:191`) does:
```
db.collection("leagues").doc(lid).update({
  memberUids: firebase.firestore.FieldValue.arrayUnion(uid),
  memberCount: firebase.firestore.FieldValue.increment(1)
});
```

This write is from the joining user, not the commissioner. Under v8 rules, the league doc update would be rejected.

**DESIGN NEEDED — 3.7.f.1.** How do we handle join writes?
- Option A: add a narrow rule carve-out: a non-member may write to `memberUids` if they are the sole difference between old and new arrays AND their uid is the one being added AND they have a valid invite or approved join request.
- Option B: move join logic to a Cloud Function (Admin SDK bypasses rules).
- Option C: keep the current array write pattern but add a join-triggered Cloud Function that validates and reverts invalid joins.

Recommendation: **Option B**. Cloud Function `joinLeague(leagueId, inviteCode)` validates invite code, validates the user isn't banned from the league, writes the join via Admin SDK. Security posture is stronger; client code simpler.

### 4.2.c — Snapshot listener behavior during mid-session role change

The existing `_memberProfileUnsub` listener at `firebase.js:517-532` replaces `currentProfile` on every doc change. In v8, if the listener fires with a change from `platformRole: "user"` to `platformRole: "suspended"`, the client must detect the transition and enforce it.

Proposed listener enhancement:

```
window._memberProfileUnsub = db.collection("members").doc(user.uid).onSnapshot(function(snap) {
  if (!snap.exists) return;
  var oldProfile = currentProfile;
  currentProfile = snap.data();
  // Detect role transition
  if (oldProfile && oldProfile.platformRole != currentProfile.platformRole) {
    handlePlatformRoleTransition(oldProfile.platformRole, currentProfile.platformRole);
  }
  // theme sync, profile bar, page re-render (existing behavior)
  ...
});

function handlePlatformRoleTransition(from, to) {
  if (to == "banned") {
    // Kick from the app immediately
    auth.signOut();  // clears currentUser; exitApp fires; user lands on ban screen via public page
    return;
  }
  if (to == "suspended" && from != "suspended") {
    window._pbSuspended = true;
    Router.toast("Your account has been suspended. You can still view content but cannot post, chat, or log rounds.");
    // Re-render current page to reflect read-only state
    Router.go(Router.getPage(), Router.getParams(), true);
    return;
  }
  if (from == "suspended" && to == "user") {
    window._pbSuspended = false;
    Router.toast("Your account has been reinstated. Welcome back.");
    Router.go(Router.getPage(), Router.getParams(), true);
    return;
  }
  if (from == "banned" && to == "user") {
    // Appeal accepted mid-session (rare — user would usually be signed out)
    Router.toast("Your appeal was successful. Reload to continue.");
    // Forced reload ensures fresh auth state
    setTimeout(function() { window.location.reload(); }, 2000);
    return;
  }
}
```

## 4.3 Suspended user flow

Per decision 3.1.a: read-only access to own profile and public content. Refinement: DMs and private chat fully locked.

### UI states during suspension

- **Home page:** renders normally with a prominent red banner at top: "Your account is suspended until [date]. [Appeal]" button opens a form (see 4.4 for the appeal form — shared UI).
- **Feed, leagues, profile views:** all render read-only. Write buttons (comment, like, DM, log round, etc.) are hidden or replaced with disabled-state text.
- **DM page:** replaced with a single-screen "DMs are unavailable during suspension" message.
- **Settings:** most options disabled. Only "view profile" and "sign out" remain accessible.
- **Shop:** visible for browsing but purchase flow disabled.

### Enforcement layers

- **Rules layer (primary):** `amIActive()` blocks every write. Critical — a user with devtools open and a crafted write cannot bypass this.
- **App layer (UX):** buttons disabled, pages rewritten. Primary purpose: don't frustrate users by letting them hit buttons that always fail with permission errors.
- **Cloud Function layer (support):** Cloud Functions check `platformRole` before acting when invoked by a user-triggered path.

### Suspension expiry

`suspension.until` is a timestamp. The system must auto-reinstate when that timestamp passes. Options:

- (a) Client-side: on app open, if `platformRole == "suspended"` AND `suspension.until < now()`, call a Cloud Function to reinstate.
- (b) Server-side: a scheduled Cloud Function fires hourly, queries `members` where `platformRole == "suspended" && suspension.until < now()`, flips them back to `"user"`.

**Recommendation: (b).** Client-side approach fails if the user never opens the app during the suspension window — they stay in the suspended state forever. Scheduled function guarantees the state flip.

**Implementation note:** Firestore doesn't support composite queries on role + timestamp without an index. Adding a composite index on `(platformRole, suspension.until)` enables the scheduled function query. Low-cost.

## 4.4 Banned user flow

Per decision 6.7.a: appeals allowed, one per 30-day cooldown. Per decision 6.5.a: email block + session invalidation on ban.

### UI states when banned user attempts to log in

1. User enters credentials.
2. `auth.signInWithEmailAndPassword` succeeds (auth remains valid because auth-side banning isn't in v8 scope — we use Firestore state to determine app access).
3. `onAuthStateChanged` fires.
4. Profile load at `firebase.js:502` succeeds, returning `{..., platformRole: "banned", ban: {...}}`.
5. `showBanScreen(currentProfile.ban)` is called instead of `enterApp`.
6. User sees:
   - Large heading: "Your account has been banned."
   - Reason (if not `reasonPrivate`): "Reason: {reason}."
   - Button: "Appeal this ban."
   - Button: "Sign out."

### Appeal flow

When user taps "Appeal this ban":

1. A form opens: single textarea, 500 char limit, "Submit appeal."
2. Submit creates a doc in a new collection `appeals/{appealId}` (implied by decision 6.7.a but not explicitly in our Section 2 list — add it).

**DESIGN NEEDED — 4.4.1.** Is there a separate `appeals` collection, or do appeals live inside `notifications` or `reports`? Recommendation: separate `appeals` collection so Founder has a dedicated queue and rate-limiting logic is cleaner.

Appeal collection schema:
- `appealId` (doc id): unique
- `fromUid`: who filed
- `filedAt`: timestamp
- `status`: `"pending"` | `"accepted"` | `"rejected"`
- `appealText`: user's argument
- `decision`: Founder's response
- `decidedAt`, `decidedBy`: Founder metadata

Rate limiting: rule denies create if `fromUid` has an appeal with `filedAt < now() - 30d` and `status != "accepted"`. Actually more practically enforced via Cloud Function — the rule alone can't easily query another doc's state.

**DESIGN NEEDED — 4.4.2.** Where does rate-limiting live — rules, Cloud Function, or both? Recommendation: Cloud Function for correctness (scanning prior appeals), rule for a basic guard (at most one appeal document per user within 30 days). Client UI also shows remaining cooldown to avoid friction.

### Session invalidation on ban

Per decision 6.5.a refinement: "email block + invalidate active auth session on ban."

Firebase Auth sessions are persistent; they don't invalidate automatically when a Firestore doc changes. Options:

- (a) Client-side detection: listener catches `platformRole: "banned"`, calls `auth.signOut()`.
- (b) Server-side via Admin SDK: ban Cloud Function calls `admin.auth().revokeRefreshTokens(uid)`, which invalidates the session on next refresh (~1 hour delay at most).

**Recommendation: both.** Client does (a) for fast UX; server-side (b) ensures the session dies even if the user never reopens the app. Client-side alone is bypassable; server-side alone has up-to-1-hour lag.

### Email block

When a banned user's account gets banned, their email is added to a blocklist. New account creation with that email is rejected at the Auth layer.

Firebase Auth doesn't have native email blocklists. Implementation:

- Maintain a `banned_emails/{emailHash}` collection (SHA-256 hashed emails for privacy).
- Intercept registration in the `validateInvite` Cloud Function (already exists per Section 1.6) — add a check: is the email's hash in `banned_emails`?
- Reject with a generic "Registration not available for this email" message.

**DESIGN NEEDED — 4.4.3.** Do we send an explicit "this email is blocked" message, or a generic error? Recommendation: generic error. Explicit messaging tells a banned user their email is the block vector, enabling easy bypass. Generic "please contact support" message forces them through the support channel, which Founder can screen.

## 4.5 Founder transfer flow

Per decisions 6.3.a and 6.8.a: gated transfer with 7-day cooling-off; revocable during cooling-off.

### State machine

```
+--------------------+    initiate    +-------------+    accept    +-------------+
| No active transfer |  ───────────►  |   pending   |  ─────────►  |  accepted   |
+--------------------+                +-------------+              +-------------+
       ▲                                   │                             │
       │ expire / cancel                   │                             │ cooling-off ends
       │                                   ▼                             ▼
       │                               +-----------+              +--------------+
       └────────────────────────────── | cancelled |              |  completed   |
                                        +-----------+              +--------------+
                                                                        │
                                                                        ▼
                                                       platformConfig.founderUid = newFounder
```

### Step-by-step

1. **Initiation (Founder):**
   - Zach opens an admin settings page: "Transfer Founder role."
   - Enters nominee's username or uid.
   - Writes a `transfer_requests/{newId}` doc with `type: "founder"`, `fromUid: zach`, `toUid: nominee`, `status: "pending"`, `expiresAt: +14d`.
   - Nominee receives a notification: "Zach has nominated you as Platform Founder. You have 14 days to accept."
   - No change to `platformConfig.founderUid` yet.

2. **Nominee accepts:**
   - Nominee taps notification, lands on transfer accept screen.
   - Sees explanation of what the Founder role entails: full platform admin, moderation responsibility, 7-day cooling-off period before the transfer finalizes.
   - Accepts.
   - `transfer_requests/{id}` updates: `status: "accepted"`, `acceptedAt: now()`, `coolingOffEndsAt: now()+7d`.
   - Zach receives notification: "{Nominee} has accepted the Founder transfer. The transfer will complete on {date} unless you revoke it."

3. **Cooling-off period (7 days):**
   - `platformConfig.founderUid` is **still Zach**. All Founder-only rules still evaluate against Zach.
   - Zach can revoke via admin panel. This updates `transfer_requests/{id}` to `status: "cancelled"`.
   - After 7 days, a scheduled Cloud Function (or the same hourly reinstate function) detects `status == "accepted" && coolingOffEndsAt < now()`. It:
     - Updates `platformConfig.founderUid` to nominee's uid.
     - Updates Zach's member doc: `platformRole: "user"`.
     - Updates nominee's member doc: `platformRole: "founder"`.
     - Updates `transfer_requests/{id}`: `status: "completed"`.
     - Sends notifications to both parties.

4. **Post-completion:**
   - Nominee now holds Founder rule authority. All `amIFounder()` checks pass for them.
   - Zach's UI — on next page load — reflects the loss of Founder capabilities.

### Interaction with founding league commissioner

Per governance 2.1, Zach is separately the commissioner of the-parbaughs. Founder transfer does NOT touch that commissioner assignment. If Zach wants to also transfer the founding-league commissioner, that's a separate commissioner transfer (see 4.6).

### Edge cases

- **Nominee's account is banned during cooling-off:** the completion Cloud Function checks `platformRole != "banned"` before finalizing. If the nominee was banned, the transfer auto-cancels and Zach is notified.
- **Zach's account is deleted during cooling-off:** blocked by decision 6.1.a — account deletion is blocked for anyone holding a role. So this can't happen cleanly. If there's an account-loss scenario (compromise, death), the out-of-band emergency flow from decision 6.3.a is invoked.

## 4.6 Commissioner transfer flow

Per decision 2.2.a: handshake flow. Future refinement: 24–48h cooling-off.

Simpler than the Founder transfer — fewer safeguards because commissioner loss is lower-stakes than Founder loss.

### State machine

Same shape as Founder transfer, with:
- `transfer_requests/{id}.type: "commissioner"`

## 5.2 New functions for v8

### 5.2.a — `onMemberRoleChange(members/{uid})`

- **Trigger:** `onWrite(members/{uid})`.
- **Fires when:** `oldData.platformRole != newData.platformRole`.
- **What it does:**

- **Trigger:** `onCreate`.
- **What it does:** Send notification to members of the affected league: "The Founder accessed private content in {league}. Reason: {reason}."
- **Cost:** N notifications for N members (~20 for the Parbaughs today).

### 5.2.c — `onLeagueDelete(leagues/{leagueId})`

- **Trigger:** `onDelete`.
- **What it does:**
  - Sweep `rounds`, `chat`, `teetimes`, `wagers`, `bounties`, `scrambleTeams`, `calendar_events`, `scheduling_chat`, `trips`, `tripscores`, `partygames`, `social_actions`, `syncrounds`, `liverounds`, `rangeSessions` — any doc with `leagueId == deletedLeagueId`.
  - Option: delete cascaded docs, OR mark them with `orphan: true` and let a scheduled cleanup handle later.
  - Delete subcollections: `joinRequests`, `moderation_log`.
  - Remove `leagueId` from every member's `leagues` array.
  - Reassign `activeLeague` for members where it was the deleted league.
- **Cost:** potentially heavy. Uses batched writes.

**DESIGN NEEDED — 5.2.c.1.** Delete cascade or mark-orphan? Marking keeps data for audit but costs storage; deleting is cleaner but irreversible. Recommendation: delete for most collections (rounds + chat are the bulk); mark `moderation_log` orphan because it's already small and sensitive.

### 5.2.d — `onErasureRequest(members/{uid}/erasure_requests/{id})`

- **Trigger:** `onCreate`.
- **What it does:** Sweep anonymization described in Section 4.7.b.
- **Cost:** heavy for power users (hundreds of docs).
- **Failure handling:** retry on transient errors; mark `status: "failed"` on permanent errors; Founder gets a notification.

### 5.2.e — Scheduled: `expireSuspensionsAndTransfers()`

- **Trigger:** `onSchedule("every 1 hours")`.
- **What it does:**
  - Query `members where platformRole == "suspended" && suspension.until < now()` → flip to `"user"`.
  - Query `transfer_requests where status == "accepted" && coolingOffEndsAt < now()` → finalize transfer.
  - Query `transfer_requests where status == "pending" && expiresAt < now()` → mark `"expired"`.
- **Cost:** small; runs hourly.

### 5.2.f — `joinLeague(leagueId, inviteCode)` (callable)

- **Trigger:** HTTPS callable from client.
- **What it does:** Validates invite code. Validates user isn't banned from the league. Adds user to `memberUids`, increments `memberCount`, writes join record. Cloud Function does this via Admin SDK, so Firestore rules don't need to allow client join writes (DESIGN NEEDED 3.7.f.1).
- **Cost:** minimal.

### 5.2.g — `initiateFounderTransfer(toUid)`, `acceptFounderTransfer`, `cancelFounderTransfer` (callable)

- Trigger HTTPS callable.
- Each validates caller authority and transitions the `transfer_requests` doc.

Similar callables for commissioner transfer.

### 5.2.h — `forceTransferCommissioner(leagueId, newCommissionerUid, reason)` (callable, Founder-only)

- **Trigger:** HTTPS callable.
- **Per decision 2.2.b:** Founder force-transfer with audit trail. Writes `moderation_log` entry on the league.

### 5.2.i — `forceDeleteLeague(leagueId, reason)` (callable, Founder-only)

- **Trigger:** HTTPS callable.

## 6.3 Rollback plan

Every irreversible change needs a backout:

### 6.3.a — If v8.0.0 migration fails mid-run

- The migration script writes to a staging collection first (`members_v8_staging/{uid}`) for each doc it processes.
- After all writes succeed, it atomically overwrites `members/{uid}` with the staged data.

- `firestore.rules` can be re-deployed with the v7.9.1 version.

### 6.3.c — If v8.2.0 `role` removal ships and breaks something

- Hardest rollback — the field is actually deleted at this point.

# Section 10 — Risks and Open Questions

## 10.1 Risk register

### Risk 1 — Migration data loss

**Likelihood:** Low. **Severity:** High.

Mitigation: staged backups (Section 6.3.a). Migration runs once; if it fails, we restore from the staging collection.

### Risk 2 — Rules rewrite introduces permission errors at scale

**Likelihood:** Medium. **Severity:** High.

Hard to catch in tests — rules that work for Zach + test fixtures may fail for specific user states. Mitigation: maintenance mode reduces blast radius; monitoring for permission-denied errors in first 48 hours.

### Risk 3 — Cloud Function cold starts add latency to auth flows

**Likelihood:** Medium. **Severity:** Medium.

Some v8 flows depend on callable Cloud Functions (joinLeague, transfers). Gen1 cold start is 1-3 seconds. Mitigation: keep critical path sync where possible; Gen2 functions for user-visible latency paths.

### Risk 4 — Founder transfer race conditions

**Likelihood:** Low. **Severity:** Critical.

A bug in the cooling-off finalization could result in no one holding the Founder role, or two people holding it simultaneously. Mitigation: Cloud Function uses Firestore transactions for the finalization step; locked `platformConfig.singleton` update; thorough testing.

### Risk 5 — Session token revocation lag

**Likelihood:** High. **Severity:** Low-Medium.

Up-to-1-hour window where a banned user still has an active session despite `revokeRefreshTokens`. Mitigation: snapshot listener catches the state change client-side; combined effect is near-immediate for active users.

### Risk 6 — Join flow via Cloud Function increases join latency

**Likelihood:** High. **Severity:** Low.

Moving join from direct Firestore write to Cloud Function adds ~200-800ms latency. Mitigation: show loading state; verify acceptable UX.

### Risk 7 — Out-of-repo Cloud Functions not version-controlled

**Likelihood:** Certain. **Severity:** Medium.

`searchCourses` and `validateInvite` are deployed but not in the repo. v8 requires modifying `validateInvite` (email blocklist). Mitigation: bring these into the repo in v8.0.0 so all future changes are tracked.

### Risk 8 — Enforcement gap during transition window

**Likelihood:** Medium. **Severity:** Medium.

During v8.0-v8.1 (legacy `role` still readable, `platformRole` also present), the transitional fallback in client code works for correctness but creates an attack surface if an attacker can influence which field is read. Mitigation: server-side rules always use `platformRole` — client-side fallback is only for display. Rules enforce correctness server-side regardless of which field the client reads.

### Risk 9 — v8.2.0 opens registration funnel without friend/handle systems

**Likelihood:** Certain. **Severity:** Low-Medium.

Leagueless users have no DM access and no handle search until v9.x lands. Mitigation: ship v8.2.0 with the "friends coming soon" messaging; monitor whether leagueless users churn out before leagues layer catches them.
