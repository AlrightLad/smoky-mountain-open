## Scalability Architecture (per Q44 Lock 3)

### Free-tier first

This is the gate item. Each option below evaluated for cost.

### Comparison matrix (3 options)

#### Option A — Branch strategy on existing production Firebase project

**Mechanism:**
- `main` branch → production GitHub Pages (`alrightlad.github.io/smoky-mountain-open`)
- New `staging` branch → secondary GitHub Pages target (`alrightlad.github.io/smoky-mountain-open-staging/` or similar via repo-level config)
- Both deploy from same Firebase project (parbaughs)
- Staging data isolated via `?league=staging-league` or feature flag

**Cost:**
- $0 — same Firebase project, same Free tier
- GitHub Pages free for public repos

**Pros:**
- Lowest setup overhead
- Code-level isolation via branch
- Easy to fold back into production (merge to main)

**Cons:**
- Data isolation is logical, not physical — staging mistakes can affect production data
- Firestore rules apply uniformly; rules changes affect both
- Shared smoke account means staging tests can pollute production data
- Cloud Functions deploy from single source; staging changes deploy to production

**10x scaling:** Staging environment doesn't scale with user count (engineering-only access); free-tier holds.

**Migration path if interim:** Move to Option B or C later if logical isolation insufficient.

#### Option B — Separate Firebase project (parbaughs-staging)

**Mechanism:**
- Create second Firebase project `parbaughs-staging`
- Staging branch deploys to it
- Full isolation: separate Auth, Firestore, Functions, indexes, rules
- Production untouchable from staging

**Cost:**
- $0 within Free tier (each project gets own free-tier quota)
- Setup time: ~2-3h (Firebase project provision, secrets sync, env vars, deploy targets)

**Pros:**
- Full data + code isolation
- Rule changes can be tested without production risk
- Cloud Functions can be deployed to staging only
- Failed migrations don't touch production

**Cons:**
- Auth doesn't mirror — staging members are separate accounts (smoke@parbaughs.test must duplicate)
- Member-facing pages render emptier in staging (no real data)
- Synchronizing schema across two projects requires discipline (migration ships double-deploy)
- API keys and config secrets doubled

**10x scaling:** Each project gets independent free-tier quota. Holds.

**Migration path if interim:** Can absorb into Option A by abandoning the second project; or evolve into Option C.

#### Option C — Hybrid: branch strategy for code, separate Firebase project for data

**Mechanism:**
- `main` and `staging` branches as Option A
- Code on staging uses production Firebase project for AUTH only (so smoke@parbaughs.test works)
- Code on staging writes to `parbaughs-staging` Firestore (separate project for data)
- Production reads/writes parbaughs Firestore only

**Cost:**
- $0 within Free tier (both projects under Free tier)
- Setup: ~3-4h (Auth federation, env-conditional Firebase init, dual config)

**Pros:**
- Full data isolation (staging writes can't touch production data)
- Member auth flow is identical (no duplicate smoke accounts)
- Highest fidelity to production while still safe

**Cons:**
- Most complex setup
- Auth-vs-Firestore project mismatch requires careful env wiring
- Risk of misconfiguration: staging accidentally writes to production Firestore

**10x scaling:** Same as Option B — independent free-tier per project.

**Migration path if interim:** Most expensive to walk back; commit deliberately.

### Recommendation (subject to Founder ruling)

**Option B (separate Firebase project)** is recommended for the following reasons:
1. Full isolation is the safest default for engineering experimentation
2. Free-tier cost = $0
3. Setup overhead is bounded (2-3h one-time)
4. Aligns with CFR Category 3 free-tier-first preference
5. Foundation for future automated staging-deploy CI/CD without production risk

Option A is faster but trades data isolation; Option C is highest-fidelity but complex.

**Founder decides at Vision authoring.**
