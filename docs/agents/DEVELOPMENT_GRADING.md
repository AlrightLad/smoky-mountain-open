# Development Grading

Self-assessment mechanism for the orchestration team. Per-wave grade with trend tracking. The system learns from its own performance, not just from Founder feedback.

## Why this exists

Per Founder directive (Vision authoring session, 2026-05-12): the orchestration team should grade its own work after every wave and track whether development quality is trending up or down over time. Grades serve three purposes:

1. **Honest self-assessment** — the team identifies its own weaknesses before Founder has to
2. **Trend visibility** — sustained downward trends warrant governance review (skills, hooks, or protocols may need amendment)
3. **Learning loop** — every error captured with root cause + prevention plan; lessons compound across waves

This is NOT a performance management system that punishes low grades. It's a feedback mechanism that surfaces patterns invisible at the ship level but obvious at the wave level.

## When grading happens

At every wave close, per INTER_WAVE_PROTOCOL.md minimum checklist. Specifically: after retrospective review of closing wave, before the lessons-learned commit, before Founder ratifies the wave gate.

Grade generated by Orchestrator + Critic jointly. Engineer contributes raw data; Orchestrator + Critic interpret.

## Grade categories

Each category scored 0-100. Final wave grade is weighted average per the weights below.

### Category 1: Ship throughput (weight: 15%)

How efficiently did the wave move?

- 90-100: All wave ships shipped within planned window
- 70-89: All wave ships shipped, some scope adjustments mid-wave
- 50-69: 80%+ of wave ships shipped, some deferred to next wave
- 30-49: Significant scope cut mid-wave
- 0-29: Wave goal missed; major restructuring needed

### Category 2: Smoke pass rate on first attempt (weight: 20%)

Per ship, did smoke pass on the first run after Engineer self-audit?

- 90-100: 90%+ ships pass smoke first-attempt
- 70-89: 70-90% first-attempt pass
- 50-69: 50-70% first-attempt pass
- 30-49: <50% first-attempt pass
- 0-29: Smoke is unreliable signal

Excludes B.43 family known-flake exception per locked governance.

### Category 3: Critic rejection rate (weight: 15%)

Per ship Critic review, how many ships needed rework before approval?

- 90-100: 90%+ ships approved on first Critic review
- 70-89: 70-90% first-review approval
- 50-69: 50-70% first-review approval (one rework cycle common)
- 30-49: <50% (two+ rework cycles common — Ship Plan template not catching enough at pre-flight)
- 0-29: Critic is gatekeeping work that should have been caught upstream

### Category 4: Inferred decision match accuracy (weight: 15%)

At retrospective, Founder ratifies or reverses inferred decisions. Match accuracy = ratifications / (ratifications + reversals).

- 90-100: 90%+ inferences match Founder intent
- 70-89: 70-90% match (pattern recognition working)
- 50-69: 50-70% match (pattern recognition needs adjustment)
- 30-49: <50% match (graduated autonomy framework needs review)
- 0-29: Inferences not matching at all; demote tier categories per GRADUATED_AUTONOMY.md

### Category 5: Member-visible regressions (weight: 20%)

Bugs that members encountered or would have encountered in production. Includes anything caught at production deploy that should have been caught at staging or earlier.

- 90-100: Zero member-visible regressions during wave
- 70-89: 1 regression, quickly resolved
- 50-69: 2-3 regressions
- 30-49: 4-6 regressions
- 0-29: Quality control breakdown

Production-affecting bugs (P0/P1) count double for this category.

### Category 6: Sanity Halt triggers (weight: 10%)

How many Sanity Halt conditions hit during the wave?

- 90-100: Zero Sanity Halt triggers
- 70-89: 1 trigger, resolved cleanly
- 50-69: 2 triggers, OR one trigger with pattern emerging
- 30-49: 3+ triggers, multiple categories
- 0-29: Sanity Halt fired routinely (governance not catching patterns)

### Category 7: Schedule adherence vs estimate (weight: 5%)

Did wave estimate hold?

- 90-100: Wave completed within 10% of estimate
- 70-89: Within 25% of estimate
- 50-69: Within 50% of estimate
- 30-49: 50-100% over estimate
- 0-29: >100% over estimate (re-estimate is needed)

Note: Founder timeline estimates are accepted as given per memory. This category grades the team's execution against the estimate, not the validity of the estimate itself.

## Final wave grade

Weighted sum. Letter grade conversion:

| Score range | Letter grade | Meaning |
|---|---|---|
| 90-100 | A | Wave executed at high quality; pattern is sustainable |
| 80-89 | B | Wave executed well; minor improvements possible |
| 70-79 | C | Wave shipped but quality issues evident; investigate |
| 60-69 | D | Wave troubled; multiple categories below target |
| 0-59 | F | Wave needs full retrospective; governance may need revision |

**Target: B+ (87) or higher on every wave.**

**Acceptable trend: flat-or-improving over consecutive waves.**

**Concern trigger: Wave grade < previous wave AND below B+ threshold.**

## Error capture format

Every error during the wave (smoke failure, Critic rejection, Sanity Halt trigger, regression, inferred-decision reversal) gets logged in the wave grade report with:

1. **What happened** (concrete observation, file path or commit hash where applicable)
2. **Root cause** (the actual reason, not the surface symptom)
3. **Prevention plan** (concrete change to skill, hook, protocol, ship plan template, or process)
4. **Severity** (P0/P1/P2/P3)
5. **Resolution** (how it was addressed in the wave)

## Development grade log

Cumulative record at `docs/agents/lessons-learned/DEVELOPMENT_GRADE_LOG.md`. Per-wave grade report committed to `docs/agents/lessons-learned/WAVE_N_GRADE.md`.

Trend analysis: at wave close, compare current wave grade to all prior wave grades. Visualize trend with simple ASCII chart in DEVELOPMENT_GRADE_LOG.md.

Sample log format:

```
Wave 1 — A- (89) — 2026-06-15
Wave 2 — B+ (88) — 2026-07-20 (slight dip; Critic rejection rate down)
Wave 3 — A (92) — 2026-09-01 (recovery; mobile-specific skills helped)
Wave 4 — TBD
```

## What grading is NOT

- Not a Founder-graded report card. Orchestrator + Critic produce. Founder reviews.
- Not a basis for agent dismissal or replacement. Agents are tools; grading evaluates the tools' performance against established protocols.
- Not a metric to game. Score inflation undermines the learning loop. If a category seems hard to score honestly, raise to Founder.
- Not a substitute for retrospective review. Grading is one input to retrospective, not a replacement.

## Trend response protocol

If wave grade trend goes negative (current wave grade < previous wave grade) AND current wave grade is below B+ (87):

1. Orchestrator surfaces the trend at wave-close retrospective explicitly
2. Founder + Orchestrator + Critic review root causes from wave grade report
3. Decide one of:
   - **Skill/hook amendment** — specific skill or hook addition or modification expected to address pattern
   - **Protocol amendment** — update to PROTOCOLS.md or other governance to catch pattern at process level
   - **Tier demotion** — if pattern reflects inferred decisions consistently missing Founder intent, demote affected category per GRADUATED_AUTONOMY.md
   - **Accept and monitor** — sometimes a wave is just hard; one bad data point doesn't warrant action
4. Action committed to lessons-learned/WAVE_N_TREND_RESPONSE.md if action taken
5. Next wave grade verifies whether action improved trend

If trend stays negative for 2+ consecutive waves below B+: this is a Founder ratification event. Pause new ship execution; do retrospective; rebuild approach.
