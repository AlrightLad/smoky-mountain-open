# BL-011: Partner-rail side-stripe — fix when `.sphud-partner` render ships

**Severity:** P3
**Captured:** 2026-05-29, v8.23.28 side-stripe coherence critique pass
**Wave/Phase target:** Whenever the spotlight partner-rail render gate ships
**Status:** Open

## Description

`.sphud-partner` (components.css) uses `border-left:3px` as a colored accent in
two ways that violate the impeccable side-stripe ban for list items:

- `.sphud-partner:hover` → `border-left-color:var(--cb-brass)` (hover affordance)
- `.sphud-partner--you` → `border-left-color:var(--cb-brass-2)` (marks current user)

This is currently **dead/forward-looking CSS** — the comment at components.css:959-963
states there is no JS consumer as of v8.14.0; the rule set "lights up" only when the
partner-rail render ships in a future gate. No member sees it today, so it is not a
live coherence violation, but it must not ship as-is.

When the partner-rail render is built, replace the side-stripe with a ban-compliant
treatment: `.sphud-partner--you` already carries `background:var(--cb-chalk-2)`, so the
"you" state reads from the background tint alone; the hover state should likewise use a
background change rather than a left rule. Also remove the `margin-left:-8px` layout
compensation that exists solely to offset the 3px transparent left border.

## Files affected

- `src/styles/components.css:977-981` — `.sphud-partner`, `.sphud-partner:hover`,
  `.sphud-partner--you`
- (future) the partner-rail render path that emits `.sphud-partner` markup

## Dependencies

- Partner-rail render gate (not yet scheduled). This item cannot be verified visually

  the v8.23.28 pass.

## Acceptance criteria for closure

- `.sphud-partner` carries no `border-left`/`border-right` > 1px colored accent
- "You" row and hover state are visually distinguished by background tint (and/or a
  leading marker), verified on a rendered partner rail at all three viewports
- No `margin-left` compensation left behind for a removed border
