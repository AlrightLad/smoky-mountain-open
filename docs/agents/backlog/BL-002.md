# BL-002: Migrate to unified loading system, then delete legacy

**Severity:** P3
**Captured:** v8.7.0 Chalk Dry primitives migration
**Wave/Phase target:** Wave 2 (blocked on screen-restructure ships)
**Status:** Open (reframed 2026-05-29 — original "delete now" premise was invalid)

## Description

Original framing ("legacy loading is dead code, delete it") is **false**. A
2026-05-29 consumer audit found the legacy `.spinner`/`.skeleton` system is
still the **live, primary** loading affordance across the app:

- `.spinner` — ~30 live consumers via the inline `<div class="loading"><div
  class="spinner"></div>…</div>` pattern (admin, admin-diagnostic, bounties,
  dms, feed, invite, leagues, members, members-detail, members-edit, range,
  partygames, richlist, scorecard, syncround, trophyroom, wagers,
  sync-attestation, router-ai-tournament, and more).
- `.skeleton`/`.skel-card`/`.skel-line`/`.skel-circle` — live via
  `skeletonCard`/`skeletonMemberRow`/`skeletonFeed` in
  `src/core/router-empty-states.js` (NOT `router.js` — the original line refs
  2199-2222 are stale; router.js was refactored to ~706 lines per AMD-027 and
  the helpers moved to router-empty-states.js). Consumed by chat.js (feed),
  dms.js + members.js (member rows).

The newer Chalk Dry loading system (`src/core/loading.js`, v8.3.4 Ship 0c —
`createLoadingState` + `skelHeader`/`skelRow`/`skelStat`/`runningPin` +
`.skel--*` CSS in components.css ~1969-2054) was **built but deliberately not
adopted**. loading.js:21 states verbatim: "No adoption in this ship — screen
ships (1-7) wire as they restructure." Those screen-restructure ships have not
happened, so the two systems coexist and the legacy one carries all traffic.

Deleting the legacy system today (the original BL-002 action) would break the
~30 surfaces above. The real work is a **migration**, with deletion as the
final step.

## Files affected

- `src/core/loading.js` — target (unified) system; already built, unadopted
- `src/core/router-empty-states.js` — legacy `skeleton*` helpers (to retire after migration)
- `src/styles/base.css` (~726-732) + `src/styles/components.css` (~57-58, ~653-663) — legacy `.skeleton`/`.spinner`/`.skel-card` CSS
- ~30 page renderers in `src/pages/` + several `src/core/*` files — inline `class="loading"`/`spinner` consumers (migrate each to `createLoadingState` / Chalk Dry primitives)

## Dependencies

- Screen-restructure ships (the loading.js "screen ships 1-7") must migrate
  pages off the legacy system first. Until then deletion is blocked.
- Founder design sign-off on the Chalk Dry loading feel before mass rollout
  (it changes the perceived loading affordance on every page — a visible change).

## Acceptance criteria for closure

- All ~30 legacy consumers migrated to `createLoadingState` + Chalk Dry primitives
- Audit re-confirms zero remaining consumers of `.skeleton`/`.spinner`/`.skel-*`
  and of `skeletonCard`/`skeletonMemberRow`/`skeletonFeed`
- Legacy CSS + helpers deleted
- Smoke + visual regression confirm no loading-state regression on migrated pages
- Ship Plan documents the migration + final audit for Critic review

## Last activity

2026-05-29 — Consumer audit during autonomous backlog triage. Original
"delete dead code" premise disproven: legacy loading is live across ~30
surfaces; Chalk Dry replacement built but unadopted. Reframed from "delete"
(P3 quick cleanup) to "migrate then delete" (design-cycle effort, blocked on
screen-restructure ships). No code changed.

## Notes

Not a quick cleanup. The deletion is trivial; the migration that must precede
it is the actual ship, and it is gated on the screen-restructure work the
Chalk Dry system was designed to ride along with.
