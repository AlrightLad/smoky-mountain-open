# BL-001: Play Now — hole par/yardage adjustment during live play

**Severity:** P2
**Captured:** v8.3.2 Honey Run hole 10 flag
**Wave/Phase target:** Wave 2
**Status:** Closed (resolved 2026-05-29, v8.23.31)

## Description

No way to adjust hole par or yardage during live play. This affects FIR (Fairway in Regulation), GIR (Greens in Regulation), and differentials. When the course pars or yardages in the database don't match the actual played course (different tees, scorecard variant, etc.), members can't correct without abandoning the round.

## Files affected

- `src/pages/playnow-scoring.js` — "Adjust" expander UI + handlers + finish payload
- `src/pages/playnow.js` — live state, deep-copied `holes` at round start
- `src/core/data.js` — `addRound` field passthrough for `holePars`/`holeYards`/`holesPlayed`
- `src/pages/caddynotes.js` — member-facing release note

## Dependencies

- W1.S4 Round capture core (canonical round shape stable) — satisfied
- Design spec for in-round par/yardage edit affordance — resolved inline (header "Adjust" toggle reveals an inline par stepper + yardage input on the hole card)

## Acceptance criteria for closure

- Member can edit hole yardage for current hole during live play
- Edits scoped to this round only (do not affect course master data)
- FIR / GIR / differential recompute correctly after edit
- Smoke covers edit flow

## Resolution (v8.23.31)

A header **Adjust** toggle (`#pn-holeedit-toggle-{hole}`) on the live hole card
reveals an inline editor: a par stepper (`#pn-holepar-val-{hole}`, clamped 3..6
with the bound buttons disabling at the edges) and a numeric yardage input
(`#pn-holeyards-{hole}`, 44px min-height, clamped 0..999). Both write to the
**round-scoped** `liveState.holes` array, which is deep-copied from the course
record at `startLiveRound` (`playnow.js:718`), so the shared course master is
never mutated. `_ensureLiveHole` materializes a dense 18-hole array on first
edit so a course with no per-hole data still yields a complete `holePars` on
finish (handicap.js skips NaN pars, which would under-count the differential).

Every commit calls `Router.go("playnow")` for a full re-render, so all
par-derived values recompute together: the top-bar running +/- (sums
`liveState.holes[i].par`), the par-3 FIR gating in `finishLiveRound`
(`_holePar(i) !== 3`), and the saved round's `holePars`/`holeYards` maps that
feed `addRound` (whitelisted passthrough added in `data.js`). The `/liverounds`
mirror written by `saveLiveState` has a fixed field set that does not include
`holes`, so per-round overrides need no Firestore-rules change.

Acceptance criteria:
- [x] Member can edit hole par for current hole during live play — `adjustHolePar`
- [x] Member can edit hole yardage for current hole during live play — `setHoleYardage`
- [x] Edits scoped to this round only — deep-copy at round start + writes confined
      to `liveState.holes`; editor copy reads "Applies to this round only. The
      course record stays unchanged."
- [x] FIR / differential recompute correctly after edit — running +/- and
      `finishLiveRound`'s `holePars`/`holeYards` recompute from the edited pars
      (verified green); `adjustedGrossScore` (handicap.js, unit-tested) consumes
      the saved `holePars`.
  - GIR note: GIR is a member-marked boolean, not par-derived, so a par edit
    correctly **preserves** the marked value rather than recomputing it. There is
    no par-driven GIR computation in the codebase to re-run.
- [x] Smoke covers edit flow — `tests/e2e/flows/09-playnow-hole-edit.spec.js`
      (5 tests, all pass): editor render, par-increase running-total recompute,
      par 3..6 clamp + stepper disabling, yardage commit + legacy `.yards` drop,
      and `finishLiveRound` carrying edited pars/yards as dense 18-length arrays.

Verified in-browser (V1) during implementation via Playwright MCP: par 4→5
recomputed the running total and the current-hole score label, yardage committed,
the par stepper clamped at 6, and the finish-map produced the expected dense
arrays. The 09-spec locks that behavior in as durable regression coverage.

## Notes

Course master data stays canonical; per-round overrides live only on the round
document's `holePars`/`holeYards`. Any downstream consumer of hole par already
reads round-override first then course master (share card `getPar`,
`updatePresence` spectator doc). If `finishLiveRound`'s payload shape changes,
re-run `09-playnow-hole-edit.spec.js` — its capture-and-halt assertion pins the
`addRound` contract.
