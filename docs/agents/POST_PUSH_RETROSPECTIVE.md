# Post-Push Retrospective Protocol

Every push the orchestration team makes produces a standardized retrospective output before declaring the ship complete. This is the Founder's window into what just happened — what changed, where we are, what agents discussed, what tests confirm, what grew.

## Why this exists

Per Founder direction (Vision authoring session 2026-05-12): "I would like to see the agent talk to one another and actually communicate not just the morse code jargon." Post-push retrospectives in plain English with explicit five-component structure ensure Founder gets meaningful visibility into orchestration team work without having to dig through commit logs, decision bubble files, or governance directories.

This is the operational layer that makes graduated autonomy actually accountable. Without it, agents could push autonomously while Founder has no visible window into what happened.

## When this fires

After **every push** to remote main. The retrospective output is generated by Orchestrator before declaring the ship complete (i.e., before transitioning to next ship or signaling readiness for next work).

The retrospective is **mandatory** — no ship is considered closed without it. Critic verifies retrospective output exists and meets the format requirements before stamp of completion.

If a ship spans multiple pushes (e.g., feature implementation push + hotfix push + caddy notes push), each push gets its own retrospective output. The retrospective is push-scoped, not ship-scoped.

## Required output components

Each retrospective output must include all five components below. Missing any component = Critic rejection, retrospective gets regenerated before ship close.

### Component 1 — What was changed

Concrete summary of what shipped, written in plain English suitable for Founder review without code context required.

Required content:
- **Files modified** with brief description of what each change accomplishes (not just file names)
- **Scope summary** in 2-3 sentences — what the push delivered as member-visible or system-visible outcome
- **Ship references** — which ship(s) this push relates to (W1.S3, M2, etc.)
- **Version impact** — semver triple-bump per locked governance (utils.js APP_VERSION + package.json + sw.js CACHE_NAME) with old → new version captured
- **Caddy Notes entry** — exact text that landed in caddynotes.js for this push (member-visible content)

Format example:
```markdown
## Component 1 — What was changed

**Ship:** W1.S3 Members directory + Find Players
**Version:** 8.22.0 → 8.23.0

**Scope summary:** Shipped the Members directory page replacing the prior member-list-with-no-search affordance, plus the Find Players cross-league friend system foundation. Founding 20 are now auto-friended as the beta baseline per locked governance.

**Files modified:**
- `src/pages/members.js` — full rewrite per spec 3a Members, A-Z alphabetical sort by username (not real name)
- `src/core/friends.js` — new module covering friend-request send/accept/decline + bidirectional relationship integrity
- `src/pages/profile.js` — public profile pages now show championship status badges
- `firestore.rules` — added friend-relationship visibility rules
- `src/pages/caddynotes.js` — version entry + member-visible description

**Caddy Notes entry shipped:**
> "Find Players is here. Browse the league directory, see who's championed what, and start adding friends across leagues. The founding crew is already pre-friended for you — no awkward first connections needed."
```

### Component 2 — Roadmap percentage and position

Where we are in the Build Roadmap (and Launch Roadmap when applicable).

Required content:
- **Current ship number / total ships** (e.g., "26 of 37 Build Roadmap ships")
- **Percentage complete** as concrete number (e.g., "70% of Build Roadmap")
- **Current wave + position within wave** (e.g., "Wave 3 / Wave 1 M1 first ship")
- **What's next** — the immediate next ship in sequence

Format example:
```markdown
## Component 2 — Roadmap position

- **Ship:** W1.S3 (Ship 3 of 37)
- **Build Roadmap progress:** 8% complete (3 of 37 ships)
- **Current Wave:** Wave 1 (3 of 19 Wave 1 ships shipped)
- **Next ship:** W1.S4 Round capture core
```

### Component 3 — Decision bubble transcripts (Plain English)

For every decision bubble that fired during this push, the Plain English Translator output gets included verbatim.

This is the heart of the new discipline — Founder sees the actual agent-to-agent conversation in language a human can read without governance-doc decoder ring.

Required content per bubble:
- **Bubble ID and short title**
- **Plain English summary** (the Plain English Translator output as drafted in DECISION_BUBBLE_AGENTS.md)
- **What we decided** + **What we considered** + **Why we chose this one** + **What we didn't choose and why** + **What this means for the codebase** + **What Founder should know** + **Key insight from the debate**

If zero bubbles fired during this push, this component reads: "No decision bubbles fired during this push. All decisions fell within current graduated autonomy tier or were governed by halt criteria."

Format example:
```markdown
## Component 3 — Decision bubble transcripts

### Bubble: friend-system-firestore-rules (20260615-1430-friend-system-firestore-rules.md)

**What we decided:** When a member sends a friend request, the receiving member sees it in their notifications but cannot view the requester's private profile content until they accept. Profile-level content stays public throughout per locked monetization model.

**What we considered:** Three options surfaced — (a) full mutual visibility from request moment, (b) one-way visibility from sender-to-receiver, (c) no additional visibility unlocks until acceptance.

**Why we chose this one:** Option (c) cleanest privacy model — no surprise visibility changes during pending state. Matches the country-club tone of "you accept the invitation, then we exchange more."

**What we didn't choose and why:** Option (a) would have created a privacy hole — sending a friend request shouldn't grant access to private content. Option (b) creates asymmetric visibility that's hard to communicate to members.

**What this means for the codebase:** Friend-request documents only contain sender ID + receiver ID + timestamp; no profile data exchanged. Profile visibility unchanged from public-by-default at launch per memory #28.

**What Founder should know:**
- This decision aligns with the public-by-default monetization model
- Doesn't preclude future paid-tier private content (those tiers govern visibility separately)
- Friend-request UI shows requester avatar + username only, no bio/stats

**Key insight from the debate:** Pending state visibility is its own category from accepted state. We need to think about this distinction whenever a "request → accept" pattern appears elsewhere (wager invites, party-game invites, league-join requests).
```

### Component 4 — Workflow document test confirmation

Per FLOW_DOCUMENTATION.md: Flow Documenter maintains `docs/flows/flows.json` and `docs/flows/flows.html` covering every member-visible flow.

Required content per push:
- **Flows changed by this push** — list of flow IDs in flows.json that were updated
- **JSON validation** — flows.json passes schema validation (no broken references, no orphaned nodes, no malformed shapes)
- **HTML rendering test** — flows.html renders correctly in browser (no broken links, no malformed markup)
- **Team confirmation** — Flow Documenter + Critic both confirm the workflow documentation is solid

If a push doesn't touch any member-visible flow, this component reads: "No member-visible flows changed by this push. Flow documentation unchanged."

Format example:
```markdown
## Component 4 — Workflow document test confirmation

**Flows changed:** flow.friend-request-send, flow.friend-request-accept, flow.friend-request-decline (3 new flows)

**JSON validation:** ✓ flows.json passes schema validation
**HTML rendering:** ✓ flows.html renders correctly across all 4 browsers
**Team confirmation:** Flow Documenter and Critic both confirm workflow documentation is solid for this push.
```

### Component 5 — Growth report

What got better as a result of this push. The compounding-improvement layer that makes the orchestration team smarter over time.

Required content:
- **Skills affected** — which skills triggered usefully, which triggered spuriously, which should be amended or retired per SKILL_PERFORMANCE_REVIEW.md
- **Bubble patterns** — what decision patterns emerged that should inform future bubble prediction
- **Governance refinements** — any halt criteria, do-NOT-halt items, autonomy tier patterns that should evolve based on this push
- **Learnings that compound forward** — what should future ships do differently because of this push
- **Pattern recognition adjustments** — Historical Pattern agent's accuracy refinement

Format example:
```markdown
## Component 5 — Growth report

**Skills affected this push:**
- `parbaughs-firestore-writer-audit` triggered usefully (3 catches, 0 spurious) — high utility, preserve
- `parbaughs-cross-surface-dependency-audit` triggered spuriously twice (matched on unrelated patterns) — propose matcher refinement at next retrospective

**Bubble patterns emerged:**
- Privacy-tier decisions (this push: friend-request visibility) follow a consistent pattern — "what visibility unlocks at which lifecycle stage." Future privacy decisions should reference this bubble as a starting point.

**Governance refinements identified:**
- None this push. Halt criteria and do-NOT-halt list held cleanly.

**Learnings that compound forward:**
- The "pending state ≠ accepted state" insight from the friend-system bubble applies to wager invites (W1.S6), party-game invites (W1.S7), league-join requests (W1.S13). When those ships fire, orchestration team should apply the same pattern analysis at design time, not retrofit later.

**Pattern recognition adjustments:**
- Historical Pattern agent should index "request → accept → visibility unlock" as a recognizable platform pattern going forward.
```

## Where retrospective output lands

Each post-push retrospective gets committed to:

```
docs/agents/retrospectives/<YYYY-MM-DD>-<short-id>-<ship-id>.md
```

Example:
- `docs/agents/retrospectives/2026-06-15-friend-system-W1.S3.md`
- `docs/agents/retrospectives/2026-06-20-bug-triage-listener-activation-W1.I1.md`

The file is created by Orchestrator, written collaboratively by participating agents (each contributing their component), and ratified by Critic before ship close.

## Format requirements

- **Markdown only** — no YAML, no JSON shoved into prose, no agent jargon
- **Plain English throughout** — written for Founder reading, not for agent re-consumption
- **No code unless essential** — code references go in component 1; component 3+ are prose
- **Brief over verbose** — each component should fit one screen scroll where possible
- **No padding** — if a component has nothing to report, the brief "no X this push" line is sufficient

## Cross-references to other governance

- **DECISION_BUBBLE_AGENTS.md** — Plain English Translator generates component 3 output
- **FLOW_DOCUMENTATION.md** — Flow Documenter generates component 4 confirmation
- **SKILL_PERFORMANCE_REVIEW.md** — Skill performance data feeds component 5
- **HALT_CRITERIA_AND_AUTONOMY_DISCIPLINE.md** — Bubble lifecycle informs component 3 cadence
- **SESSION_JOURNAL.md** — Per-session decisions feed components 3 + 5
- **DEVELOPMENT_GRADE_LOG.md** — Wave-close grades reference patterns from retrospectives
- **CTO_INTERFACE.md** — Retrospectives are the primary Founder-visibility surface between Vision authoring and wave-close grading

## Critic verification

Before ship close, Critic verifies:
1. All five components present and non-empty (or "no X this push" lines if applicable)
2. Component 1 references match actual commits (no fabricated file lists)
3. Component 2 percentages match roadmap ground truth
4. Component 3 decision bubble references match files in `docs/agents/decision-bubbles/` and lessons-learned archive
5. Component 4 workflow assertions match actual flows.json + flows.html state
6. Component 5 growth observations have specific evidence (not generic "things improved" platitudes)

If verification fails, Critic rejects, Orchestrator regenerates, ship doesn't close until clean retrospective lands.

## What this enables

This protocol gives Founder:
- **Visibility into agent reasoning** without reading the bubble files directly
- **Roadmap awareness** at every push, not just at wave close
- **Growth tracking** that compounds across pushes
- **Quality verification** via component-level Critic gating
- **Pattern recognition** by reading retrospectives across ships

This protocol gives orchestration team:
- **Forced reflection** before declaring work done
- **Standardized output** that future agents can read consistently
- **Skill performance feedback loop** without manual prompting
- **Bubble pattern catalog** that strengthens future bubbles

## Activation

This protocol activates at **Phase 1 commit** (governance lands in docs/agents/).

First retrospective output fires at first push after Wave Zero Dry-Run passes. Wave Zero Dry-Run itself produces a retrospective per this protocol — that retrospective is the proof the protocol works before real ships depend on it.

## Audit cadence

- **Every push** — retrospective generated and Critic-verified
- **Every retrospective close** — Founder reviews; ratifies or red-lines
- **Every wave-close** — meta-review: are retrospectives surfacing patterns Founder is acting on? Are they reading well? Component-level refinements happen here.
- **Every Build → Launch transition** — comprehensive protocol review; component additions/refinements considered before Launch Phase A begins.
