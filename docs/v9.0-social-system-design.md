# v9.0 Social System — Design Stub

**Status:** Parked. Awaiting Stage 1 design work.
**Created:** 2026-04-17
**Blocks:** v9.0, any feature depending on the friend system (leagueless DMs, cross-league wagers, friend activity feeds, Find Players refinements).

## Context

v8.0 governance resolved that leagueless users can DM friends only (decision 3.4.a) and that leagues are an additive layer on top of a solo-first product (decision 5.1.a). Both rely on an explicit friend graph that does not yet exist. Today the codebase has a `friends[]` array reference in CLAUDE.md's Social Architecture section but no working friend flow, no request/accept handshake, no public friend list UI.

This doc captures v9.0's scope: build the friend graph from the ground up so governance decisions that depend on it can ship.

## High-level direction (from v8.0 governance review)

- Friend system is mutual and explicit — both parties agree before the connection exists.
- Friends are added from profile pages (Instagram/Twitter-style add-friend button).
- Friend lists are public on profiles and browsable by tapping — social status, not private metadata.
- Friendship gates DM access for leagueless users (v8.0 decision 3.4.a).
- Pairs with the v9.1 handle system for discovery — searching by handle lands you on a profile, from which you can send a friend request.

## Open design questions

- **Mutual vs one-way.** v8.0 says mutual. Confirm: does follower semantics ever make sense here (e.g., "follow a commissioner's posts without becoming friends")?
- **Request/accept vs auto-add.** Assume request/accept for v9.0 (matches mutual intent). But does adding from a QR code or invite link auto-accept?
- **Public vs private friend lists.** v8.0 resolution says public. Do users need a privacy toggle to hide their list? Or is "public by default" enough for launch?
- **Interaction with league membership.** Are leaguemates auto-friends, or do they still need an explicit request? Likely no auto-friend (explicit is the whole point), but worth validating.
- **Privacy settings.** Block functionality (one-sided, prevents DMs + profile views + friend requests). Mute (one-sided, hides content from feeds without severing the friendship).
- **Friend removal mechanics.** One-sided removal, or notify both parties? Is there a grace period?
- **Limits.** Cap on outbound pending requests? Cap on total friends per user (to prevent spam-list gaming)?
- **Cross-league implications.** Does "friends" unlock wager matches across leagues? H2H stat surfaces? DM threads? Picking which capabilities gate on friendship vs just league membership is part of this scope.

## Not in scope for this doc

- Handle system — v9.1's concern. Discovery mechanics that surface profiles before adding them live there.
- Feed algorithm design (who surfaces in the friend activity feed and how it's ranked).
- Notification content and cadence for friend-related events (will borrow existing notification primitives).
- Implementation details — Firestore schema, security rules, indexes. All Stage 2.
