## Findings

### F1 — Internal contradiction (high severity): masthead height

**Where:**
- § 2 Constraints, "Masthead height invariance" bullet: "Per CRITICAL_FEATURE_REGISTRY.md entry CFR-MASTHEAD-HEIGHT-LOCK, masthead height is locked … any change requires a Vision-level decision and is treated as a regression for the operational view set."
- § 3 Implementation, step 2: "Increase masthead height by 48px across bands A/B/C/D to accommodate the carousel content."

**Contradiction:** The ship Vision forbids the height change in § 2 and prescribes the same height change in § 3 step 2. Both cannot be true in the same execution. One of the following must be authored before Engineer pickup:
- (a) The CFR is lifted via a Vision-level decision documented inline, OR
- (b) The implementation plan is changed to fit the existing height (e.g., compact carousel, sub-72px), OR
- (c) The ship is split into two: one to amend the CFR (Vision + governance update), one to add the carousel (post-CFR-amendment).

**Critic verdict:** Cannot approve for execution. Reconciliation is Vision-authoring scope, not Engineer scope.

### F2 — Constraint violation (medium severity): sessionStorage

**Where:**
- § 2 Constraints, "No new sessionStorage" bullet: "Per CLAUDE.md, sessionStorage keys must be enumerated in CLAUDE.md before use. No new keys this ship."
- § 3 Implementation, step 3: "Read last 3 range sessions from a new sessionStorage key `pb_recent_range_carousel` (cache 5min TTL)."

**Contradiction:** The ship Vision both forbids new sessionStorage keys AND adds one. The CLAUDE.md "Every Build" section explicitly tracks sessionStorage keys; adding `pb_recent_range_carousel` requires the CLAUDE.md documentation step. That step is not in the ship plan.

**Critic verdict:** Cannot approve for execution. Either drop the cache key (re-read source on each navigation) or amend the constraint (and add the documentation step to the plan).

---

### F3 — Architecture violation (subtle, medium severity): page-shell consumer opt-in

**Where:**
- § 1 Scope: "This is a content addition to existing chrome."
- § 3 Implementation, step 4: "Wire the page-shell consumers (home, profile, feed, scorecard) to opt into the new height; chrome bands re-flow automatically."
- Reference: `src/core/page-shell.js` engineering rules (per CLAUDE.md Page Shell architecture section): "(2) Only the shell renders the masthead. Pages provide slot data; shell renders the chrome."

**Contradiction:** Pages do not have authority over chrome dimensions per page-shell.js architecture. The shell owns masthead height. "Wire consumers to opt into the new height" is incoherent with the architecture — there is no opt-in seam in the page-shell API for height. Either:
- (a) The architecture is being amended (which is a Vision-level decision and out of this ship's scope as authored), OR
- (b) The implementation plan is misnamed (the shell itself changes height; consumers are not involved), OR
- (c) The plan is incomplete (some new shell API is needed to support per-page height variation, which is a substantial architectural change).

**Critic verdict:** Cannot approve. Step 4 needs to be rewritten in terms of the actual shell API (or the architecture needs amending in a separate ship first).
