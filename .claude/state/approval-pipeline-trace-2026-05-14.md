# Approval pipeline trace — 2026-05-14

Per Founder direction "APPROVAL PIPELINE LOCKDOWN — Make decisions
auto-propagate". This document traces the current end-to-end flow,
identifies the failure mode behind the 7+ stalled approvals observed
during this session, and proposes the fix.

## Current pipeline (Option B — Downloads-watcher pickup)

Founder's machine already runs the Downloads-watcher pattern (Option B
in Founder's framing). The flow:

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. proposals.html UI                                            │
│    Founder clicks Approve / Reject / Defer on a proposal row.   │
│    UI accumulates decisions in localStorage.                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. Founder clicks "Export decisions"                            │
│    UI generates decisions-<ISO-ts>.json with shape:             │
│      { kind: "decisions",                                       │
│        decisions: [ { id, decision: "approved"|"rejected"|     │
│                       "deferred", note: "..." }, ... ] }        │
│    Browser auto-downloads → %USERPROFILE%\Downloads\             │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. PARBAUGHS-Downloads-Watcher (Scheduled Task, every 5 min)    │
│    scripts/cron/downloads-watcher.ps1                           │
│    a. Preflight: tree clean? If dirty + non-routine, SKIP.      │
│    b. Scan Downloads for decisions-*.json, amendments-*.json,   │
│       escalations-*.json newer than marker.                     │
│    c. For each: route by `kind` field → apply script.           │
│    d. apply-decisions.sh moves proposal from                    │
│       .claude/state/proposals/pending/ → approved/ (or          │
│       rejected/ deferred/).                                     │
│    e. Update marker. Run regen-all.ps1.                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 4. PARBAUGHS-Proposal-Readiness-Scanner (every 2h)              │
│    scripts/cron/proposal-readiness.ps1                          │
│    Scans approved/ for proposals ready to ship (per their       │
│    own ship-criteria). When ready: triggers ship-close →        │
│    moves approved/ → shipped/.                                  │
└─────────────────────────────────────────────────────────────────┘
```

The same SKIP pattern persisted for hours. The pipeline's preflight gate

**The actual artifacts that pile up during normal agent work include:**

Note that `scripts/visual-audit/enumerate-interactives.mjs` appearing in
the dirty list at 21:30 is actually a tracked-file edit — that one is
NOT routine and was correctly held back; main agent later committed it.

The genuine routine items (user-context PNG captures, heartbeats,
deferred markers, pycache) are what cause the recurring SKIP.

## Proposed fix (Option B refinement — widen AMD-020 allowlist)

Founder's directive frames this as Option A vs Option B. The Founder's
own note "Most likely current architecture per existing cron. Verify it
works." correctly identifies Option B as the in-place pattern.

The fix is NOT to rebuild the pipeline. The fix is to widen the auto-
commit allowlist in `downloads-watcher.ps1` AND in any matching `maintenance.ps1`
+ `proposal-readiness.ps1` preflight checks. Specifically:

```powershell
# REVISED — covers actual agent-work output categories
$routinePatterns = @(
    # Pre-existing telemetry
    '^\.claude/state/telemetry/events/.+\.ndjson$',
    '^\.claude/state/telemetry/aggregates/.+\.json$',
    '^package-lock\.json$',

    # NEW — user-context + audit outputs (Class A artifacts per AMD-020)
    '^\.claude/state/main-flows-v2/founder-real-context/.+\.(png|json)$',
    '^\.claude/state/main-flows-v2/iter-.+\.png$',
    '^\.claude/state/user-journey-audits/.+\.(png|md|json)$',
    '^\.claude/state/heartbeats/.+\.json$',
    '^\.claude/state/dashboard-health/.+\.ndjson$',
    '^\.claude/state/security/cycles/.+\.ndjson$',
    '^\.claude/state/proposals/ship-readiness-deferred/.+\.json$',

    # NEW — Python bytecode (also gitignore candidate)
    '^scripts/__pycache__/.+\.pyc$',
    '^tests/__pycache__/.+\.pyc$',

    # NEW — proposal/amendment/escalation inbox processing artifacts
    '^\.claude/state/proposals/inbox/.+\.json$',
    '^\.claude/state/amendments/inbox/.+\.json$',
    '^\.claude/state/escalations/inbox/.+\.json$',
    '^\.claude/state/proposals/\.last-processed-decisions\.json$'
)
```

In parallel, add the cache + browser session paths to `.gitignore`:

```
# Python bytecode
scripts/__pycache__/
tests/__pycache__/
**/*.pyc

# Playwright MCP session (browser-control)
.playwright-mcp/
```

## Verification

Per Founder direction, build `scripts/verify-approval-pipeline.sh`:

1. Snapshot current `approved/` count.
2. Drop a synthetic `decisions-<ts>.json` (test proposal already
   created in `pending/test-pipeline-canary.md`) into `~/Downloads/`.
3. Wait up to 6 minutes for the watcher to fire.
4. Assert `approved/test-pipeline-canary.md` now exists.
5. Assert `approved/` count == snapshot + 1.
6. Assert tree clean post-run.
7. Clean up canary (move back to pending/, remove from approved/).

Test/QA agent runs this on every cycle. Pipeline regression = automatic
detection.

## Why not Option A (direct file write)

Option A requires either a local backend, the Capacitor wrapper, or
the File System Access API. All three are heavier than the existing
Option B and add a moving piece. The current Option B pipeline is
sound — it's the allowlist that's too narrow. Widening the allowlist
costs one file edit; Option A would cost a new server or wrapper.
