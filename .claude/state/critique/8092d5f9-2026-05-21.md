### F1 (HIGH) — docs/walkthroughs/staging-firebase-project.md, .claude/state/task-queue/founder/staging-firebase-project.md

Walkthrough embeds literal Firebase Web SDK config (apiKey, appId) in heredoc Founder pastes into `.env.staging`. While Firebase Web API keys ARE public-by-design (they identify the project, not authenticate — Firestore rules + Cloud Function security own the auth boundary), the pattern sets precedent: the next Founder following this template will paste a Sentry DSN (genuine secret) or Cloud Function runtime key the same way. Also AMD-018 gate #3 "Auth provider config" requires Founder pre-authorization for any auth-provider enable; the walkthrough instructs Email/Password toggle without an explicit pre-auth record.

**Fix applied:**
- Moved literal API key + App ID into `<details>` block with explicit "this is a public Firebase Web key, not a secret" annotation
- Replaced inline heredoc values with `<copy-from-firebase-console>` placeholders + linked to the source
- Added explicit AMD-018 gate #3 pre-auth scope to task-queue/founder/staging-firebase-project.md: "Founder authorizes Email/Password provider only; OAuth/SMS/Phone require separate pre-auth"

### F2 (HIGH) — scripts/founder-mark-complete.ps1

`Invoke-Expression $verifyCmd` is PowerShell's `eval`. Three concrete problems:
1. `Bash(eval *)` was added to deny list in same ship — built the verify mechanism on the exact arbitrary-code-execution pattern the security policy now bans
2. Front-matter `verify_command` parsed by simple regex that strips one quote layer; `$(rm -rf .claude)` in a verify_command would execute on Founder paste
3. `$LASTEXITCODE` carries over from previous external command; pure-PowerShell pipelines like `if (Test-Path) { Select-String } else {'NOT_FOUND'}` inherit a stale code → script flags `verified-closed` even when Select-String returns nothing

**Fix applied:**
- Replaced `Invoke-Expression` with `& powershell -NoProfile -NonInteractive -Command $verifyCmd` (child-process scope)
- Added allowlist of permitted commands at top of verify (Test-Path, Select-String, Get-Content, firebase, git, npm, Get-ChildItem); rejects with clear error if verify_command starts with anything else
- Reset `$LASTEXITCODE = 0` before invoke; also check `$?` in addition to exit code
- Added explicit length cap on verify_command (200 chars) to prevent payload smuggling

### F3 (MEDIUM) — scripts/regen-founder-checklist.py + scripts/founder-mark-complete.ps1

Front-matter YAML parsing disagrees between Python regen and PowerShell mark-complete:
- Python uses flat-key regex per-line, strips single layer of quotes
- PowerShell uses multi-line regex with lookahead `(?=\r?\n[a-z_]+:|\r?\n---)` — TRUNCATES verify_command at first colon-followed-by-letters

Sentry verify dodges this by coincidence; the next Founder writing `Select-String '^something:'` gets silently truncated.

**Fix applied:**
- Documented the front-matter contract at top of regen-founder-checklist.py: "Flat key:value pairs only. No multi-line values. No colon-followed-by-letters in values (use spaces or escape)."
- Replaced PowerShell parser with: extract the literal `---...---` block, then parse line-by-line with the SAME regex Python uses (`^([A-Za-z_][A-Za-z0-9_]*):\s*(.*)$`)
- Both scripts now share identical parse semantics

### F4 (MEDIUM) — scripts/sidecar/usage-snapshot.ps1, quota-status.json schema

`Get-AutoDerivedWeeklyTokens` returns last-7-days rolling sum but writes it into `status.weekly_tokens` — same field that elsewhere carries calendar-week and claude.ai-anchored values. P9 (data truthfulness) violation: Founder cannot tell from surface alone whether `weekly_tokens=5.5B` is week-to-date, rolling-7d, or anchored to claude.ai's reset boundary.

**Fix applied:**
- Added `weekly_window_basis: "calendar-week" | "rolling-7d" | "claude-ai-anchored"` to quota-status.json schema
- Auto-derived branch sets `weekly_window_basis="rolling-7d"`
- Manual-paste branch sets `weekly_window_basis="claude-ai-anchored"` (claude.ai shows %used since its own reset boundary)
- No-data branch sets `weekly_window_basis=null`
- Updated `_warning` to call out the basis explicitly

### F5 (LOW) — scripts/founder-mark-complete.ps1

`Out-File $statePath -Encoding utf8` writes UTF-8 *with* BOM on Windows PowerShell 5.1. Python's `read_text(encoding="utf-8")` does NOT strip BOM → raises `json.JSONDecodeError` on `﻿{` → silently falls back to empty state. After first mark-complete run, dashboard shows stale state forever.

**Fix applied:**
- Replaced `Out-File` with `[System.IO.File]::WriteAllText(..., (New-Object System.Text.UTF8Encoding($false)))` (no-BOM)
- Same pattern usage-snapshot.ps1 already uses correctly
- For defense-in-depth, changed `regen-founder-checklist.py:load_state` to `encoding="utf-8-sig"`
