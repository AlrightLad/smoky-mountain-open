## Findings + applied fixes

### F1-pass2 (HIGH) - regex double-escape in verify_expected silently breaks verify

**location:** `.claude/state/task-queue/founder/staging-firebase-project.md:6`

The verify_expected was authored as `"env-ok[\\s\\S]*parbaughs-staging"`. After
the shared front-matter parser strips one layer of double quotes, both Python
and PowerShell see the literal string `env-ok[\\s\\S]*parbaughs-staging`. PS
`-match` and Python `re.search` then treat `\\s` as literal-backslash-then-s,
NOT whitespace. End-to-end: Founder follows walkthrough, .env.staging exists,
firebase command emits `env-ok\r\nparbaughs-staging  (parbaughs-staging)`, but
the regex fails to match. P9 violation: "verification-failed" surfaces without
tracing to actual truth.

**Fix applied:** Changed to `"env-ok[\s\S]*parbaughs-staging"` (single-escape).
YAML treats `\s` literally inside double-quoted strings (no shell-style
expansion), so the regex consumer sees the intended `\s`.

### F2-pass2 (MEDIUM) - walkthrough Step 4 contradicted F1's narrow gate#3 pre-auth scope

**location:** `docs/walkthroughs/staging-firebase-project.md:107-116` vs
`.claude/state/task-queue/founder/staging-firebase-project.md:15-21`

The task-queue .md narrowed AMD-018 pre-auth to gate #3 ONLY (Email/Password
provider). But walkthrough Step 4 said "agent runs `firebase deploy --only
firestore:rules --project staging`" -- that's gate #1 (Firestore rules deploy),
which is NOT pre-authorized by this task. A Founder following the walkthrough
verbatim hits Step 4, expects the agent to deploy, and the agent must refuse
(gate-protected.sh blocks rules edits + gate #1 hook blocks deploy). Internal
contradiction.

**Fix applied:** Rewrote Step 4 to "Founder runs these directly in PowerShell"
with explicit commands + note that a separate pre-auth file would be needed to
delegate this to the agent. Preserves the narrow gate#3 scope F1 established.

### F3-pass2 (LOW) - mojibake purge from F2 was incomplete

**location:** `scripts/founder-mark-complete.ps1`, `scripts/regen-founder-checklist.py`, `scripts/sidecar/usage-snapshot.ps1`

The previous critique advertised "smart-quote mojibake purge converted em/en
dashes + smart quotes to ASCII". Audit found 1 + 8 + 6 = 15 non-ASCII bytes
remaining across the three .ps1 / .py scripts -- `->` arrows, em-dashes, middle
dots. PowerShell 5.1 reads UTF-8 only when explicit `-Encoding utf8` is passed
at every entry point; mojibake risk persists for any caller missing the flag.

**Fix applied:** Ran complete ASCII-only sweep on all three files. Replaced
arrows (`->`), em/en-dashes (`--`), middle dots (`-`), smart quotes (`"` `"`
`'` `'`), and ellipsis (`...`). Re-validated parse-ok on both PowerShell
(`[Parser]::ParseFile`) and Python (`ast.parse`). One regex side-effect
collateral: `extract_title` regex character class `[\-=-:]` (after mojibake
replacement) emitted Python FutureWarning about set difference at position 21;
collapsed to `[-:]` since both old chars (em-dash, ASCII hyphen) reduced to
hyphen.
